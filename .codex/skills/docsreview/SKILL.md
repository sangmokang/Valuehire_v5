# Docs Review

## 목적

`docs/` 하위 문서의 중복, 상충, 낡은 정의, 잘못된 위치, 끊어진 참조를 반복 점검한다. 발견과 제안은 읽기 전용으로 수행하고, 내용 선택·병합·이동·보관·삭제는 사장님의 명시적 승인을 받은 항목에만 적용한다.

## 필수 원칙

- 감사 단계에서는 어떤 문서도 수정하지 않는다.
- `docs/README.md`의 문서 지도와 SOT 규칙을 먼저 따른다.
- 명시된 SOT/ADR/소유권 규칙은 비교 근거이지 무단 정리 권한이 아니다.
- 파일명, 수정 시각, 문서 길이, 더 최신 날짜만으로 정본을 결정하지 않는다.
- 중복 또는 상충 문서 중 하나를 임의로 택하거나 삭제하지 않는다.
- 사용자의 기존 미커밋 변경을 되돌리거나 덮어쓰지 않는다.
- 승인 없는 무인·주기 실행은 보고서 출력 후 반드시 멈춘다.
- 승인은 해당 실행에서 제시한 파일 지문에만 유효하다. 승인 후 파일이 바뀌면 다시 제안한다.

## 1. 사전 점검

1. 저장소 루트의 `AGENTS.md`, `docs/README.md`와 대상 하위 경로의 `AGENTS.md`를 읽는다.
2. 제품 범위 판단이 필요하면 `docs/product/shared/service-build-plan.md`의 관련 절만 읽는다.
3. `git status --short -- docs .codex/skills/docsreview .claude/skills/docsreview .gitignore`로 작업 트리를 확인한다.
4. 대상 범위가 없으면 `docs/` 전체를 대상으로 하되, 한 번에 최대 10개의 우선 검토 항목만 승인 요청에 올린다.
5. 파일 목록, 제목/헤딩, 명시적 SOT 표기, 내부 링크, 동일 내용 해시를 수집한다. 의미 비교가 필요한 후보는 본문을 직접 읽는다.

## 2. 읽기 전용 감사

다음 유형으로 분류한다.

- `EXACT_DUPLICATE`: 내용이 동일한 복제본
- `NEAR_DUPLICATE`: 목적과 핵심 정의가 실질적으로 겹치는 문서
- `CONTRADICTION`: 같은 대상에 양립할 수 없는 규칙·수치·상태를 선언한 문서
- `STALE_OR_SUPERSEDED`: 폐기 또는 대체 근거가 문서 안에 명시된 문서
- `MISPLACED_OR_ORPHANED`: `docs/README.md`의 위치 규칙과 어긋나거나 참조되지 않는 문서
- `BROKEN_REFERENCE`: 존재하지 않는 로컬 파일·앵커를 가리키는 참조

단순히 표현이 다르거나 서로 다른 시점의 기록이라는 이유만으로 충돌로 분류하지 않는다. 회의록, 작업 로그, 증거 스냅샷은 현행 규범 문서와 역할이 다를 수 있으므로 보존 목적을 먼저 확인한다.

## 3. 승인 패키지

각 후보에 안정적인 ID `DR-YYYYMMDD-NN`을 붙이고 아래 형식으로 보고한다.

| ID | 유형 | 문서와 근거 | 충돌/중복 요약 | 권고 | 선택지 | 영향 |
|---|---|---|---|---|---|---|

근거에는 클릭 가능한 `path:line`과 현재 Git blob 해시 또는 `shasum -a 256` 앞 12자를 포함한다. 선택지는 상황에 맞게 다음 중 제시한다.

- `A 유지`
- `B 유지`
- `병합안 작성`
- `둘 다 유지 + 역할 명시`
- `이동/보관`
- `삭제`
- `보류`

권고와 이유는 제시하되 결정을 대신하지 않는다. 승인 질문은 "승인할 ID와 선택지"를 요청한다. 사용자가 명시적으로 고른 ID가 하나도 없으면 편집 단계로 넘어가지 않는다.

## 4. 승인 후 적용

1. 승인된 각 ID의 대상 파일을 다시 읽고 지문을 재검증한다.
2. 지문이나 관련 미커밋 변경이 달라졌으면 해당 ID를 `재검토 필요`로 돌리고 수정하지 않는다.
3. 승인된 선택만 최소 변경으로 적용한다. 승인되지 않은 인접 문제는 별도 후보로 남긴다.
4. 이동·삭제 전 `rg`로 모든 참조를 찾아 함께 수정할 범위를 제시된 영향과 대조한다.
5. 병합 시 선택된 의미를 보존하고, 필요한 경우 대체 문서에 `Supersedes` 또는 역할 설명을 명시한다.
6. 문서 구조가 바뀌면 `docs/README.md`의 지도만 필요한 만큼 갱신한다.

승인이 모호하면 추정하지 말고 읽기 전용 상태로 멈춘다. `전체 정리`, `알아서 정리`는 개별 상충 내용의 선택 승인으로 간주하지 않는다.

## 5. 검증과 보고

적용 후 최소한 다음을 수행한다.

```bash
git diff --check
git diff -- docs/ .codex/skills/docsreview/ .claude/skills/docsreview/ .gitignore
```

이동·삭제·링크 수정이 있었다면 `rg`로 이전 경로의 남은 참조와 새 경로의 존재를 확인한다. 최종 보고는 다음 네 항목으로 끝낸다.

1. `Summary`: 승인 ID별 실제 처리 결과
2. `Files Changed`: 실제 변경 파일만
3. `Verification`: 실행 명령과 결과
4. `Assumptions / Risks`: 보류된 충돌, 재검토 필요 항목, 자동 검증 한계

## 주기 실행 계약

- 호출 예: `$docsreview docs/ 전체를 읽기 전용으로 감사하고 승인 패키지만 제시해.`
- 매 실행은 현재 파일을 기준으로 처음부터 재검사하며 과거 승인이나 보고서를 자동 재사용하지 않는다.
- 새 문제가 없어도 검사 범위와 검증 방법을 보고한다.
- 스케줄러에서 호출된 실행은 승인 패키지만 남기고 문서를 변경하지 않는다. 사용자가 같은 대화에서 명시적으로 승인한 뒤에만 별도 적용 단계를 실행한다.
