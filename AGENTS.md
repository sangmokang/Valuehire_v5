# AGENTS.md — Codex 작업 시 지켜야 할 규칙

이 문서는 Codex(그리고 이 레포에서 일하는 다른 AI 에이전트)가 따라야 할 규칙을 담는다.
코드 작업 절차는 `docs/harness.md`(표준 작업 루프)를 SOT로 따른다.

---

## 모든 사용자 브리핑(중간·차단·최종) 규칙 — 항상 적용

사장님께 보이는 **모든 사용자 브리핑**에 적용한다. 작업 중간 업데이트, 멈춤·오류·차단
보고, 최종 결과를 포함한다.

- 중간·차단 보고는 1~3문장으로 **현재 상태 / 이유 / 다음 행동**만 말한다.
- 최종 보고는 아래 5칸짜리 "쉬운 보고서" 형식으로만 말한다.
- 내부 상태 근거와 사용자 설명은 **1:1로 대조**한다. 근거를 빼거나 과장하지 않고,
  확인한 기술 상태 하나마다 사장님이 이해할 수 있는 쉬운 문장 하나로 번역한다.
- 스킬·프롬프트가 기술 보고 형식이나 영어 제목을 요구해도 내부 기록에만 사용한다.
  사장님께 보이는 메시지에는 이 쉬운 보고 규칙이 우선한다.
- 기술 용어·내부 절차명(RED/GREEN, verdict, worktree, SOT, 게이트 번호,
  파일 경로:줄번호 등)은 절대 그대로 쓰지 않는다. 꼭 필요하면 괄호 안에 쉬운 한 줄을 붙인다.

```
## 1. 뭘 요청받았나 (한 문장)
사장님이 원한 것을 그대로 한 문장으로.

## 2. 뭘 했나 (사람 말로, 불릿 3~5개)
- "무엇을 고쳤다/추가했다"를 결과 중심으로.
  예: "학력·회사·학교 세 가지 점수를 더 세밀하게 나눠 매기도록 고쳤습니다."
- 숫자·기준은 남기되 왜 중요한지 한 줄 덧붙인다.
  예: "박사까지 나온 사람은 10점, 학사는 5점처럼 단계별로 나눴습니다."

## 3. 확인은 어떻게 했나 (검증 — 쉬운 말로)
- "검사를 몇 개 돌렸고 몇 개 통과했는지"만 숫자로.
  예: "관련 검사 23개, 전체 검사 1,485개 — 전부 통과했습니다."
- "두 번 검증했다"는 이렇게: "제가 한 번 확인하고, 다른 도구로 한 번 더
  교차 확인했습니다. 둘 다 이상 없었습니다."
- 기술 용어(RED/GREEN, mutation 등)는 절대 그대로 쓰지 않는다.

## 4. 놓친 부분 / 다음에 봐야 할 것 (있으면)
- 범위 밖이라 안 건드린 것, 애매해서 남은 리스크를 사람 말로.
- 없으면 "특별히 걸리는 부분은 없습니다"라고 명시.

## 5. 다음 행동 (사장님이 뭘 해야 하는지)
- "올려도 됩니다 / 승인 기다립니다 / 확인 부탁드립니다" 중 명확히 하나.
- push·PR·merge 여부와 그 이유를 한 줄로.
```

톤: 합쇼체(정중체), 비즈니스 보고서 톤. 문장은 짧게, 번역투 금지.
"~했습니다", "~입니다"로 끝맺는다. 이모지·느낌표 남발 금지.

> 이 규칙은 사용자에게 보이는 보고 문체에 적용된다. 코드·커밋 메시지·verdict.json 등 내부 산출물의
> 형식과는 무관하다 — 내부적으로는 지금처럼 RED/GREEN·게이트 번호를 그대로 써도 된다.
