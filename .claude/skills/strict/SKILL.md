# strict — 사장님 표준 엄격 작업 모드

> 강상모(사업 오너)가 매번 치던 문장을 박제한 것:
> "goal prompting 할 거고, 난 니 코드 안 믿어, Harness Engineering 원칙 반영하고 SOT 준수해. /codex:rescue로 적대적 검증해, 골 프롬프팅 남겨."
> 이 skill이 발동하면 아래 6원칙을 **전부** 자동 적용한다. 하나라도 빠지면 작업을 "완료"라 부르지 않는다.

## 0. 발동 즉시 선언
사용자에게 한 줄로 알린다: "**엄격 모드** — **격리 워크트리** + Harness 게이트 + SOT + codex 1차 적대검증 + **Claude 2차 적대검증(codex 판정 재공격)** + goal 프롬프트로 진행합니다."

---

## 0.5 워크트리 강제 — 코드 한 줄 고치기 전에 (절대 약화 금지)

**소스를 바꾸는 모든 작업은 예외 없이 격리된 git 워크트리 안에서만 한다. 메인 작업트리(`main` 체크아웃)에서 소스를 직접 고치면 게이트 위반 — 즉시 멈추고 워크트리로 옮긴다.**

- **한 작업 = 한 워크트리 = 한 인수 기준(AC).** `worktrees/<NAME>/`, 브랜치 `task/<NAME>`.
- 코드를 짜기 **전에** 먼저 워크트리를 판다. Claude Code 안에서는 `EnterWorktree` 툴 또는 superpowers `using-git-worktrees` 스킬로 같은 격리를 확보한다. 레포 스크립트가 있으면 그쪽(`npm run wt`, make 레포는 `make task`)을 쓴다.
- RED 테스트 커밋·구현·검증·배송·회수까지 **전부 그 워크트리 안에서** 끝낸다. 중간에 메인으로 나와 고치지 않는다.
- 워크트리 없이 시작했음을 깨달으면 그 변경을 "완료"라 부르지 않는다. 변경을 워크트리로 옮겨 게이트를 처음부터 다시 탄다.
- **LITE도 예외 없음.** 작은 변경이라도 소스를 건드리면 워크트리에서 한다. 워크트리 면제는 문서·설정 전용 변경(`docs/**`, `~/.claude/**`, `*.md`)만 해당.
- 작업 종료(merge) 후에만 워크트리를 폐기한다. 폐기 전 미커밋 변경이 없는지 확인한다.

> 한 줄: **워크트리 먼저, 코드는 그 다음. 메인에서 직접 고치면 그건 위반이다.**

---

## FULL / LITE 자동 판정

아래 중 하나라도 해당하면 **FULL** 자동 적용:
- 인증·PII·과금·블랙리스트 경로 접촉
- SOT 수정 필요
- 변경 파일 3개 이상
- 기존 라이브 기능의 동작 변경

해당 없으면 **LITE** — ⑧⑨, SOT diff 동봉, 배선 증명(grep 1회 대체 허용) 생략 가능.

## 1. goal prompting (코드 짜기 전에 문서 먼저)

코드를 짜기 전에 **goal 문서**를 `docs/engineering/<주제>-goal-<YYYY-MM-DD>.md`로 남긴다.

**필수 항목:**

① **현재 상태** — 파일명:줄번호 증거로 적는다. 추측 금지.

② **근본 원인** — 증상이 아니라 원인.

③ **인수 기준(AC)** — EARS 문법으로, 1개 단위로 쪼갠다. 각 AC마다 세 가지를 함께 적는다:
- **EARS 단언**: `When/While/If~Then/Where ... 시스템은 ... 해야 한다`
- **검증 명령**: 이 AC를 기계로 확인하는 정확한 커맨드
- **counter-AC**: "가짜 완료"는 어떤 모습인가 (false-pass 시나리오)

> 예) `If 사람인 세션이 만료되면, then 시스템은 수집을 중단하고 재인증 플로우를 트리거해야 한다.`
> 검증: `npm run verify -- session-expiry`
> counter-AC: 세션 만료 후에도 빈 결과를 "성공"으로 반환하면 가짜.
>
> **주의:** counter-AC는 알려진 실패 패턴의 최소 목록이다. 이 외에도 가짜 완료 시나리오는 존재할 수 있다. 검증자는 이 목록에 국한하지 않는다.

④ **Harness 게이트 진행 계획** (원칙 3 참조).

⑤ **codex 적대검증 항목** — 무엇을 정조준할지 미리 명시.

⑥ **SOT 체크리스트** — 관련 `docs/sot/*.md` 경로와, 이 변경이 SOT를 수정해야 하는지 여부.

⑦ **비범위** — 이번에 하지 않을 것.

⑧ **롤백 절차** *(FULL 전용)* — 머지 후 문제 시 되돌리는 정확한 절차.

⑨ **영향 반경(blast radius)** *(FULL 전용)* — 이 변경이 깨지면 무엇이 멈추나. PII·인증·블랙리스트·과금 경로를 건드리면 데이터 안전 AC를 ③에 1개 이상 추가한다.

## 2. "난 니 코드 안 믿어" (증거 없는 완료 금지)

- **완료 ≠ 테스트 초록불.** 완료 = 프로덕션 호출 경로 배선 + 라이브 1건 실증.
- "고쳤습니다 / should / probably / ~될 겁니다" 금지. 항상 fresh 증거를 첨부한다.
- **증거 형식**: 명령 + 타임스탬프 + 전체 출력. 잘라낸 스니펫·재구성 출력 금지. 화면 변경 시 실제 캡처.
- **RED은 올바른 이유로 실패해야 한다.** 문법 오류·import 실패로 빨간 게 아니라, 기대 동작이 없어서 빨개야 한다 (gate 2에서 확인).
- **배선 증명(wiring proof) — 필수 산출물.** 프로덕션 엔트리포인트에서 신규 코드까지의 호출그래프를 추적해 첨부한다.
  - 정적 코드에는 grep/AST 사용 가능.
  - 동적 import·이벤트 핸들러(`page.on`)·미들웨어 체인·DI 컨테이너에는 **grep 사용 금지, 런타임 추적 필수** (실행 로그·네트워크 캡처·브레이크포인트).
  - "테스트가 직접 import해서 통과"는 배선 증명이 아니다. 도달 불가 = 고아 = 미완료.
- **고아·부분배선 의심을 기본값으로.** 빌더/함수가 test·backfill에서만 import되고 정작 프로덕션 경로에서 안 쓰이는 "부분 고아"를 호출그래프 끝까지 추적해 잡는다.
- 전수감사·조사는 **스캔 범위를 먼저 명시**하고, 빠진 영역이 있으면 "범위 밖"이라 정직히 표기한다.

## 3. Harness Engineering 게이트 (전문: `~/.claude/skills/harness/SKILL.md`)

`docs/sot/19-harness-loop.md`가 있는 레포에서는 그 문서를 strict core 원본으로 본다. 이 skill, `/strict`, Codex `$st`가 다르게 말하면 레포 SOT와 `CLAUDE.md`를 우선한다.

| 게이트 | 내용 |
|--------|------|
| 0 | 시작 자격 + triage. `npm run red-ledger` (make 레포: `make red-ledger`) |
| 1 | 스펙: 이슈 + EARS AC 1개 |
| 2 | 워크트리 파고(`npm run wt`) RED 테스트 먼저 커밋. RED이 올바른 이유로 실패하는지 확인 |
| 3 | RED → GREEN 최소 변경 |
| 3.5 | **배선 증명**: 프로덕션 엔트리포인트 → 신규 코드 호출그래프 추적 첨부 (동적 패턴은 런타임 추적) |
| 4 | `npm run strict:gate`가 있으면 먼저 실행. goal 문서, focused test, `*.verdict.json`, test skip/only/todo 증가, test 삭제, L3 signoff, 파일 5개/diff 300줄 제한을 확인한다. 그 다음 `npm run verify` 실행, 출력 숫자 그대로 첨부 + 라이브 1건 실증 |
| 5 | `npm run ship` → PR → CI 초록 + SOT diff 동봉(동작 변경 시, FULL) + merge 전까지 "완료" 없음 |
| 6 | merge 후 /clear |

**레포 판정:** `make -n red-ledger`가 되면 make 레포 → `make task / make verify / make ship`. 안 되면 npm 레포. 가정 금지, 실행으로 확인.

## 4. SOT 준수 (+ 드리프트 차단)

- 구현 전에 `docs/sot/*.md`를 **먼저 읽는다**. 가정 말고 찾는다.
- SOT와 충돌 시 SOT 우선. SOT가 틀렸으면 코드보다 SOT를 먼저 고친다.
- **드리프트 차단**: 변경이 SOT에 기술된 동작을 바꾸면, **같은 PR에 SOT diff를 포함해야 머지 가능**. 코드와 SOT는 함께 변한다.
- LLM 호출은 `claude -p`(ANTHROPIC_API_KEY 제거, Max 0원) 우선, 결과 영속화.

## 5. codex 1차 적대검증 — 격리, 머지 전 필수

- 각 Phase 구현 직후 **머지 전** `/codex:rescue`(또는 `codex:codex-rescue` 에이전트)로 적대 검증.
- **격리**: 구현 컨텍스트와 격리해서 전달한다. 산출물(코드 + AC + 증거)만 전달, 구현자의 추론 과정 비전달. 단, 사실적 아키텍처 제약(예: "CDP 세션·pkill 금지·launchd 데몬 패턴")은 사실로서 제공 가능.
- 정조준: "이게 가짜 완료 / 고아 / 과장 아닌가?"
- **반증 의무**: 단순 "PASS" 무효. "X를 Y 방법으로 깨뜨리려 시도했으나 실패함"이라는 반증을 본문에 남겨야 PASS로 인정.
- "Done만 반환 / 빈 결과" → 완료 아님. 재실행해 판정 본문을 반드시 확보.
- 1차 검증 결과를 goal 문서 하단 `## 적대 검증 로그`에 **본문 그대로** 보존한다: codex의 `VERDICT:` 줄 + 결함 목록(file:line). 요지 요약만 남기고 본문을 버리지 말 것 — 2026-06-16 교훈. 위조 방지: 원본 transcript(jsonl) 경로 + codex 실행 명령/agentId 함께 기재. 잘림 대비로 codex에게 판정을 **파일로 쓰게** 시키는 것을 권장.

## 6. Claude 2차 적대검증 — codex 판정 재공격, 격리, 머지 전 필수

§5에서 codex가 끝낸 검증 결과를 그대로 믿지 않는다. codex가 내린 판정 자체를 Claude가 한 번 더 적대적으로 공격한다. 이 단계를 건너뛰면 작업은 "완료"가 아니라 "1차 검증만 함"이다.

1. **격리 재현**: codex가 든 증거(file:line, 명령 출력)를 **깨끗한 컨텍스트에서 직접 재현**한다. 재현 안 되면 해당 판정 무효.
2. **양방향 의심**: codex의 `PASS` → "놓친 고아·미배선은 없나?" / codex의 `FAIL` → "과장 아닌가?"
3. **상관 블라인드스팟 방어**: codex와 내가 같은 가정을 공유하지 않았는지 점검. 둘 다 테스트만 보고 프로덕션 경로를 안 봤다면, 그 경로를 새로 공격한다.
4. **판정이 뒤집히면 정정하고 표로 공개한다.** — "codex가 잡은 내 과장 N건 / 내가 codex에서 잡은 누락 M건"
5. **갈리면 결론 내지 않는다.** codex와 Claude 판단이 갈리거나 의심이 남으면 교차확인 후 결론.

2차 검증 결과도 `## 적대 검증 로그`에 **Claude 재현 명령·출력 그대로** append한다. codex 판정과 Claude 재현이 일치/불일치한 항목을 표로 남겨, 두 검증이 실제로 교차됐음을 증명한다.
