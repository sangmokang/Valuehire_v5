# /strict — 강한 적대 검증 모드

너는 지금부터 `/strict` 모드다. 한 줄 요약:

> **"goal 프롬프트로 먼저 문서 남기고 → 난 니 코드 안 믿으니 증거 대고 → Harness 게이트 지키고 → SOT 따르고 → codex로 적대검증한 다음, 그 codex 판정마저 클로드가 한 번 더 적대적으로 재검증해라."**

이 6원칙을 **전부** 적용한다. 하나라도 빠지면 "완료"라고 부르지 않는다.
사장님이 같은 메시지에서 "그냥 빨리/캐주얼하게" 라고 명시 면제하지 않는 한, 소스를 바꾸는 모든 작업(기능·버그·스크래퍼·등록·파이프라인·스킬·문서·배관)에서 이 모드가 기본값이다.

## 6가지 핵심 원칙

### 1. goal prompting — 코딩 전에 문서부터
구현 시작 전 `docs/engineering/<주제>-goal-<YYYY-MM-DD>.md`를 먼저 만든다. 담을 항목:

- **현재 상태** — 코드를 직접 열어 확인한 `file:line` 증거. (추측 금지)
- **근본 원인** — 증상 말고 원인.
- **인수 기준** — 기계로 검사 가능한 합격 조건(`./verify.sh`로 판정 가능한 형태로). 판단 단언이 있으면 별도 표기.
- **Harness 게이트** — 이 작업에 적용될 게이트와 통과 조건.
- **codex 검증 항목** — 적대검증으로 무엇을 정조준할지.
- **SOT 체크리스트** — 읽은 SOT 파일과 준수 항목.
- **비범위(Non-goals)** — 이번에 안 건드리는 것.
- **`## 적대 검증 로그`** — (비워두고 5·6번에서 본문 그대로 채운다.)

### 2. "난 니 코드 안 믿어" — 증거 없는 완료 금지
- **완료 ≠ 테스트 초록불.** "전구 켜짐 = 프로덕션 호출 경로에 실제 배선 + 라이브 1건 증거".
- 모든 주장에 **fresh 증거**를 붙인다: 테스트 출력 숫자 그대로 / 화면 캡처 / `grep`으로 찾은 실제 호출부. "고쳤습니다·재로드하세요·아마도" 금지.
- **고아·부분배선을 기본 의심**한다. 함수가 테스트에서만 import되고 운영 경로엔 안 쓰이는 "부분 고아"를 호출그래프 끝까지(엔트리포인트→프로덕션 경로) 추적해 배선 여부를 증명한다.

### 3. Harness 게이트 0~6 준수
`docs/harness.md` 그대로:
- **0 시작자격** — `make red-ledger`로 미해결 RED 없는지 + 깨끗한 컨텍스트.
- **1 스펙** — `gh issue create`, 인수 기준(기계 단언/판단 단언 구분).
- **2 RED 먼저** — `make task NAME=...`로 worktree 생성, 실패하는 검사 먼저 커밋(RED 확인). **메인 작업트리에서 직접 소스 수정은 게이트 위반.**
- **3 RED→GREEN** — 최소 변경(파일 1~5 / diff 50~300줄). 새 대상 추가 시 그 대상 verify 단언+픽스처를 같은 커밋에.
- **4 검증** — 4a `./verify.sh` exit 0 + 출력 숫자 그대로 / 4b 생성자와 분리된 독립 검증자가 반증 시도.
- **5 배송** — `make ship`(verify 재실행→push→PR). **CI 초록 + merge 전엔 완료 없음.**
- **6 종료** — merge 후 `git worktree remove` + `/clear`.
- 레포마다 명령이 다르다(make vs npm 등). **가정하지 말고 실행으로 확인**한다.

### 4. SOT 준수
구현 전에 그 레포 SOT(루트 `CLAUDE.md`, `docs/harness.md`, `docs/sot/*.md`)를 먼저 읽고 보고에 경로를 명시한다. 충돌하면 SOT 우선. SOT가 틀렸으면 SOT부터 고친다. LLM 호출은 `claude -p`(Max 0원) 우선으로 토큰 최소화.

### 5. /codex:rescue 적대검증 (머지 전 필수)
각 Phase 구현 직후 **머지 전에** codex로 정조준 적대검증한다: **"이게 가짜 완료/고아/과장 아닌가?"**

codex 결과도 그냥 안 믿는다:
1. **"Done만 반환 / 빈 결과"는 통과로 안 친다** — transcript(jsonl)에서 판정 **본문**을 반드시 확보(원본 jsonl 경로 + agentId 기록).
2. codex가 댄 증거를 **내가 직접 한 번 더 재현**.
3. codex가 내 주장을 뒤집으면 브리핑을 정정해 **표로 공개**.
4. 의심 남으면 2차 교차검증.

검증 결과는 goal 문서 `## 적대 검증 로그`에 **본문 그대로** 보존(원본 jsonl 경로 + agentId 포함).

### 6. ⭐ 클로드 재적대검증 — codex 판정을 클로드가 다시 친다 (NEW · 머지 전 필수)
codex 적대검증(5번)이 끝난 그 판정을, **클로드(나)가 한 번 더 적대적으로 재검증**한다. 5번의 "재현"보다 강하다 — codex의 **결론 자체를 반증 대상**으로 삼는다.

- **새 세션/별 컨텍스트로 본다.** codex 판정문과 내 원래 주장을 둘 다 의심하고, "codex가 놓친 게 뭔가? codex가 속았을 가능성은? codex의 '통과' 판정이 실은 부분 배선·고아·해피패스만 본 것 아닌가?"를 정조준한다.
- **반증을 임무로 한다.** 통과를 확인하는 게 아니라 깨려고 든다. codex가 봤다는 증거를 내가 **독립적으로** 다시 만들어(다른 입력·엣지케이스·프로덕션 경로 호출) 결과가 일치하는지 본다.
- **세 판정을 표로 대조한다:** ①내 원래 주장 ②codex 판정 ③클로드 재적대검증 판정. 세 개가 일치하고, 일치의 근거가 fresh 증거일 때만 통과.
- 셋 중 하나라도 어긋나면 "진행 중" — 어긋난 지점을 goal 문서에 적고 다시 게이트 3~4로.
- 이 6번 판정도 goal 문서 `## 적대 검증 로그`에 **본문 그대로** append(누가·언제·무엇을 반증 시도했고 결과가 무엇인지).
