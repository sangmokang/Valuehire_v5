자연어 문장을 /fleet-run으로 바꿔주는 로직과, 그 뒤 실제 큐에 넣는 로직만 따로 떼어 자동 검사를 방금 다시 돌렸습니다. 83개 검사 전부 통과했습니다.

pytest tests/test_hermes_fleet_bridge.py tests/test_hermes_plugin_registration.py tests/test_fleet_dispatch.py tests/test_fleet_worker.py

확인된 것

  • ClickUp 링크만 줘도 인식
  • "사람인", "잡코리아", "win" 같은 한글 단어 인식
  • "/aisearch 링크", "/humansearch 링크"도 인식
  • 관계 없는 대화·링크는 무시(오작동 방지)
  • 보낸 사람이 등록 안 된 사람이면 거부

이전 점검 기록: 잡 22는 정상 Windows AI Search가 아니었습니다

아래는 오늘(2026-07-13) 이전 시점에 실 운영 Discord 봇을 직접 확인했던 기록입니다. 이번 재검증은 코드 자체만 다시 돌린 것이라, 잡 22가 지금 어떤 상태인지는 이 기록과 별개로 /fleet-status로 다시 확인이 필요합니다.

job 22 · queued machine: macmini skill: humansearch params: {}

올바른 목표값은 machine=winpc, skill=aisearch, channels=[saramin,jobkorea], execution=live입니다.

macmini heartbeat는 약 61시간 44분 오래됐고 winpc heartbeat는 없습니다. 따라서 잡 22는 현재 자동으로 시작될 상태가 아닙니다.

잡 22를 기다리거나 같은 요청을 반복 전송하지 마세요. 먼저 아래 취소 명령으로 정리하는 것이 안전합니다.

/fleet-cancel job:22

가장 쉬운 사용법

Hermes v5 봇과 개인 DM을 엽니다

서버 공개 채널이 아니라 hermes_v5 앱과의 개인 대화에서 입력합니다.

ClickUp 링크와 win을 한 메시지로 보냅니다

사람인과 잡코리아를 모두 검색하는 가장 안전하고 명확한 입력입니다.

https://app.clickup.com/t/9018789656/86ey35mg3 사람인 잡코리아 win

접수 JSON의 네 필드를 확인합니다

machine=winpc, skill=aisearch, 두 채널, status=queued가 보여야 합니다.

배포 전 주의

현재 실제 Hermes gateway에는 새 자연어 라우터가 적용되지 않은 것으로 확인됩니다. 운영 배포가 끝나기 전에는 위 문장도 구 배선으로 처리될 수 있습니다. 응답 JSON이 기대값과 다르면 즉시 중단하고 반복 전송하지 마세요.

정상 접수 응답 판별법

아래 핵심 모양과 같아야 합니다. job id와 시각은 매번 달라집니다.

{
  "action": "enqueued",
  "job": {
    "machine": "winpc",
    "skill": "aisearch",
    "position_url": "https://app.clickup.com/t/...",
    "params": {
      "channels": ["saramin", "jobkorea"],
      "execution": "live"
    },
    "status": "queued"
  }
}

정상

  • winpc
  • aisearch
  • 원하는 채널 목록
  • queued

잘못 접수

  • ClickUp인데 macmini
  • humansearch
  • params: {}
  • 같은 메시지에 응답 2개

아직 시작 전

  • queued는 접수만 된 상태
  • started_at: null
  • heartbeat가 stale이면 대기 고착
  • worker claim 후 running

“애플리케이션이 응답하지 않았어요”가 보일 때

같은 요청을 바로 다시 누르지 않습니다

Discord 응답만 늦고 서버에는 이미 잡이 만들어졌을 수 있습니다. 반복하면 중복 잡 위험이 있습니다.

/fleet-status를 한 번 실행합니다

방금 시각의 잡이 있으면 원 요청은 접수된 것입니다. machine·skill·params까지 확인합니다.

잘못된 queued 잡이면 취소합니다

/fleet-cancel job:<잘못된 job id>

운영자에게 다음 네 가지만 전달합니다

  • 보낸 원문 한 줄
  • 표시된 job id
  • 응답의 machine / skill
  • 메시지를 보낸 시각

“Unknown command”와 enqueue JSON이 같이 오면

같은 Discord 봇 토큰으로 구 gateway와 새 gateway가 동시에 응답하는 상태입니다. 입력 문제가 아니라 운영 gateway 중복 문제이므로 더 보내지 말고 운영자에게 알립니다.

aisearch와 humansearch 구분

aisearch — 지금 필요한 기능

  • 입력: ClickUp 포지션 링크
  • JD를 읽고 검색어·동의어 생성
  • 사람인 OR/AND/NOT 직접 입력
  • 잡코리아 키워드 칩 직접 입력
  • 기본 머신: winpc

humansearch — 다른 기능

  • 이미 만들어진 검색 결과 화면 순회용
  • 검색어 생성·필터 입력 기능 아님
  • LinkedIn 기존 검색 URL 등에 사용
  • ClickUp 링크만 주는 새 검색에는 사용하지 않음
  • 잡 22가 잘못 들어간 이유를 보여주는 신호

운영 준비가 끝났는지 확인하는 기준

현재는 아래 세 항목이 미완료입니다

  1. 실제 Hermes 호스트에 최신 valuehire_fleet 플러그인 symlink·활성화
  2. winpc의 Supabase 응답 및 stale하지 않은 heartbeat
  3. Chrome connector 또는 승인된 전용 CDP 프로필

Hermes

  • gateway 정확히 1개
  • 플러그인 load 오류 0
  • 자연어 → fleet-run rewrite
  • 동일 message id 중복 0

Windows worker

  • task Running
  • heartbeat 2분 이내
  • machine=winpc
  • 고정 keepalive Disabled

브라우저

  • 사람인 로그인 marker 정상
  • 잡코리아 로그인 marker 정상
  • captcha/2FA 우회 없음
  • 외부 발송 0건
