# Goal 프롬프트 — 전체 코드 재리뷰 (Harness Engineering · 불신 전제)

> **언제 쓰나:** 사장님이 `/goal` 또는 "전체 코드 다시 리뷰해" 라고 하실 때.
> **무엇을 하나:** 저장소 전체를 "내 코드는 헛점이 있다"는 전제로 다시 훑어,
> 결함을 **검사(RED)로 먼저 고정**하고, 작은 단위로 고치고, **두 번 적대적으로 깨고**,
> 하니스 게이트로 배송한다.
>
> 이 프롬프트는 SOT 불변식 5(불신 → 2패스 적대검증 → 과거 지시 회수)와
> `docs/harness.md` 게이트 0~6 위에서만 돈다. 임의 리뷰가 아니다.

---

## 0. 절대 원칙 (이 리뷰 내내 약화 금지 — SOT 불변식)

1. **3사 자동 로그인을 막지 않는다.** 로그인 경로를 "막는" 변경은 제안조차 하지 않는다.
2. **사장님이 크롬 쓰는 게 보이면 무인 작업은 즉시 멈춘다(R4 양보).**
3. **제안·메일 자동 발송 금지.** 리뷰·수정 중 어떤 코드도 outreach를 **자동 발송**하게 만들지 않는다 — 보내기는 항상 사람이 누른다.
4. **사장님께는 쉬운 한국어로 보고한다.** (게이트 4a 숫자는 "검사 N개 통과"처럼 풀어서)
5. **내 코드는 믿지 않는다 — 두 번 깐다. 시작 전 과거 지시부터 찾는다.**

---

## 1. 시작 전 — 과거 지시 회수 (게이트 0.5, 건너뛰기 금지)

리뷰를 시작하기 전에, 사장님이 **전에 시킨 것**과 **이미 있는 것**을 먼저 찾는다.
"새로 짜기" 전에 "이미 있나?"를 묻는다. 최소 3축:

- **메모리:** `~/.claude/projects/*/memory/MEMORY.md` 와 항목 파일에서 과거 지시·결정 확인.
- **기존 코드:** `grep -ri "<키워드>" tools/ scripts/` , `git log --oneline --all` 로 같은 기능·과거 작업 사례 확인.
- **스킬·문서:** `~/.claude/skills/` 와 `docs/`, 루트 `*.md` 에서 같은 절차가 이미 명문화됐는지 확인.

→ 회수 결과를 한 줄로 남긴다: "기존 사례: 있음(경로) / 없음". 있으면 **재사용·확장**으로 방향을 잡는다.

---

## 2. 리뷰 범위 — 작은 기능 단위로 쪼갠다

전체를 한 덩어리로 보지 않는다. 작은 단위로 나눠 각 단위를 독립적으로 깬다. 예시 축:

- **포털 자동화** (`tools/multi_position_sourcing/`): 로그인·세션 지속·검색·재시도·백오프·점유 양보(R4).
- **AI Search / 저수지** (`embed.py`, `match.py`, `scoring.py`, `grouping.py`): 점수 rubric, 근거 URL 실존, 임베딩.
- **개인정보·안전장치**: 금지 연락처·민감정보 수집 차단, secret 로그 노출, 자동 발송 차단.
- **스킬·문서 계약**: SKILL frontmatter/트리거, 문서 근거 경로 실존.
- **배관**: `Makefile`·`verify.sh`·pre-push 훅·CI(`verify.yml`)의 멱등·재현성.

각 단위마다 아래 3절(찾기 → RED → 두 번 깨기)을 **독립적으로** 돈다.

---

## 3. 각 단위 절차 — TDD + 2패스 적대검증

이 저장소의 리뷰는 "읽고 의견 내기"가 아니라 **검사로 고정하기**다.

### 3.1 결함을 RED로 먼저 고정 (TDD)
- 의심 가는 동작·주장을 **실패하는 테스트**로 먼저 쓴다(`make task NAME=...` 워크트리에서).
- "고치면 GREEN, 일부러 깨면 RED" 가 되는 단언만 인정한다. 막연한 우려는 단언으로 못 쓰면 버린다.
- `main`은 읽기 전용 — 모든 변경은 워크트리에서.

### 3.2 최소 수정으로 GREEN
- RED→GREEN 최소 변경(파일 1~5 / 50~300줄). 새 문제는 고치지 말고 새 RED로 분리.
- 자기확장: 새 대상 추가 시 그 대상의 verify 단언 + 픽스처를 같은 커밋에.

### 3.3 두 번 깐다 (게이트 4b · SOT 불변식 5)
- **패스 1 — 자기 적대검증(스스로 반증):** 내가 직접 내 수정본을 공격한다.
  빈 값·잘못된 입력·경계값·막힌 사이트(429/captcha)·중복·동시성·secret 노출을 일부러 넣어 깨본다.
- **패스 2 — Codex Rescue 독립 2차검증:** 같은 수정본을 `codex:rescue`(Codex Rescue)에게 넘겨
  **다른 눈으로** 한 번 더 적대적으로 깬다. Codex Rescue가 반례를 찾으면 GREEN 아님 → 3.1로.
- 두 패스 모두 못 깨야 통과. 그 전엔 사장님께 "됐다"고 보고하지 않는다.

---

## 4. 검증 게이트 (기계 판정)

- `./verify.sh` 실행 → exit 0 == GREEN. 출력 숫자(`N passed, M failed`)를 증거로 그대로 본다.
- exit 0 아니면 "진행 중". "고쳤습니다/재로드하세요" 금지.
- 사장님 보고는 풀어서: "검사 N개 다 통과했어요. 이상 없습니다."

---

## 5. 배송 + 진화

- 단위별로 `make ship`(verify 재실행 → push → PR). PR 본문: 이슈 + 증명 테스트 + Codex Rescue 판정 + verify 출력.
- merge 후 워크트리 정리 + /clear.
- **진화:** 게이트를 통과했는데 나중에 결함이 새어나오면, 그 결함을 잡는 RED를 먼저 추가한 뒤 고친다.
  Codex Rescue가 반복해서 같은 결함을 잡으면 그 검사를 4a 기계검사로 승격한다.

---

## 6. 최종 보고에 포함할 것 (쉬운 한국어)

- 훑어본 단위 목록과, 단위별 "기존 사례 있음/없음"(과거 지시 회수 결과).
- 찾은 결함 → 고정한 RED → 고친 내용 → **자기 적대검증 / Codex Rescue 결과**.
- `./verify.sh` 실제 출력(만들어내지 말 것 — 실패면 실패라고).
- 남은 RED(미해결) 목록과 다음 작업 제안.

> 주의: 실행하지 않은 검사 결과를 지어내지 않는다. 자동 발송은 어떤 경우에도 하지 않는다.
