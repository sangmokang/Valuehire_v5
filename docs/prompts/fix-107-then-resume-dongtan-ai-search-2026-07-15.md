## 1단계 — 이슈 #107 완료 상태를 읽기 전용으로 검사한다

2. main 저장소에서 `git fetch origin main`을 실행하되 파일은 쓰거나 버리지 않는다. 다음 완료 조건을
   각각 실제 명령과 파일로 확인한다.
   - `gh issue view 107`: 상태가 `CLOSED`
   - `gh pr view 117`: 상태가 `MERGED`
   - `git merge-base --is-ancestor a090640 origin/main`: 성공
   - `.harness/red-ledger.tsv`: #107 행이 `GREEN`
3. 완료 증거와 함께 아래 계약을 실제 코드·문서·검사에서 읽기 전용으로 1:1 대조한다.
   - 사장님이 기기나 크롬을 쓰는 동안 자동 작업을 멈춘다.
   - 마지막 이상 신호 뒤 180초 동안 문제가 없으면 자동으로 이어간다.

5. SOT25의 1200초·최소 300초는 후보 검색 중 `OCCUPIED` 상태를 다시 확인하는 예약 간격이고,
   최상위 규칙의 180초는 사장님이 손을 뗀 뒤 작업자가 자동 재개하는 기준이라 서로 다른 범위다.
   다만 두 문서가 같은 양보·재개 흐름을 다르게 표현한 문서 간 어긋남은 #107 완료 여부와 별개로 기록한다.

6. 완료 조건과 읽기 전용 대조가 모두 맞으면 저장소 파일·작업 장부를 수정하지 않고, 기존 보조 작업
   공간을 하나도 재사용하지 않으며, 새 브랜치·커밋·PR을 만들지 않은 채 즉시 2단계로 간다.

   2단계로 간다. 단순 표시 차이만으로 이미 끝난 구현을 다시 만들지 않는다.

## 2단계 — 원래 화성·동탄 시스템운영팀 AI Search를 재개한다

사용자 승인 범위는 후보 검색과 차감 없는 프로필 저장이다. ClickUp·Discord 등록과
제안·메일·InMail 발송은 별도 요청이 없으므로 실행하지 않는다.

### 입력 JD

- 포지션: 시스템운영팀 IT 인프라 / 보안, 2명
- 근무지: 화성·동탄
- 경력: 관련 경력 5년 이상, 대리~과장급
- 학력: 관련 학과 4년제 대학 졸업 이상
- 핵심: 서버·스토리지·네트워크 설계/구축/운영, 온프레미스·AWS, 장애 원인 분석,
  백업·복구, 성능·용량 계획, 보안 정책·취약점·로그 분석
- 도구: FortiGate, NAC, NetClient, Gradius DLP, Shell Script, Python
- 우대: 제조업·규제 산업, 자격증, AI 아키텍처, 문서중앙화
- LinkedIn 시작 URL:
  https://www.linkedin.com/talent/hire/1764960900/discover/recruiterSearch?searchContextId=fc984960-726b-4339-af08-4b1216a17b55&searchHistoryId=21274719372&searchRequestId=4bfa26d4-847d-4ce5-96a9-0f37cb7fb442&start=0&uiOrigin=FACET_SEARCH

### 실행 순서

1. `$ai-search` 스킬과 SOT25를 읽고 정본 검사기를 실행한다. 이 단계는 저장소를 고치는 작업이 아니라
   로그인된 포털의 라이브 후보 검색이다. 코드 작업용 미완료 장부나 코드 검사 환경 오류만으로
   라이브 후보 검색을 중단하지 않는다. 정확한 검색 실행 경로가 실제로 깨졌을 때만 해당 경로를 막는다.
2. 브라우저를 조작하기 전에 실행 중인 모든 디버그 크롬의 실제 포트, 전용 프로필, 열린 탭,
   로그인·캡차·다중세션 상태를 확인한다. 사용자가 로그인을 마쳤으므로 다시 계속 여부를 묻지 않는다.
3. 채널별 상태를 확인하고 준비된 사람인·LinkedIn을 함께 진행한다. 실제 캡차·2FA·
   다중세션 차단이 생기면 해당 채널만 멈추고 나머지는 계속한다.
4. 사람인은 인재풀에서 화성·동탄, 경력 5년 이상, 4년제 조건으로 추가 검색한다.
   인프라 운영, 시스템·네트워크, 정보보안, AWS·자동화 조합을 각각 확인한다.
5. LinkedIn은 제공 URL의 프로젝트·경력·위치 필터를 유지한다. 키워드는 반드시 좌측 필터 패널의
   `Keywords → Profile keywords or boolean`에 넣고, 상단 전역 검색창은 사용하지 않는다.
6. LinkedIn 위치 필터에 화성·동탄 또는 화면에서 제공되는 가장 가까운 정식 위치가 적용됐는지
   확인한다. 수원·용인·오산 등으로 넓히면 별도 확장 검색으로 표시해 조용히 섞지 않는다.
7. LinkedIn 키워드는 최소 네 갈래로 다변화한다.
   - 인프라 중심: IT Infrastructure / Infrastructure Engineer / System Engineer / 인프라
   - 네트워크·보안 중심: Network Security / Security Engineer / 정보보안 / FortiGate / NAC / DLP
   - 제조업 중심: manufacturing / 제조 / factory + infrastructure / security / network
   - AWS·자동화 중심: AWS / on-premise / Linux + Python / Shell Script / automation
8. 결과 수에 맞춰 키워드를 좁히거나 넓히고, 상세 프로필은 근거 화면과 함께 저장한다.
9. 후보는 직무 직접성, 학력, 이직 안정성, 제조업·도구 경험으로 평가한다. 아래 네 필드가 모두
   있는 후보만 추천한다: `profile_url`, `score`, `why_fit`, `profile_summary`.
10. 제안·메일·InMail은 발송하지 않는다. 실제 사람 개입이 필요한 차단 외에는 “계속할까요?”를
    묻지 말고 다음 단계로 이어간다.

- 내부 상태 근거 하나마다 쉬운 사용자 문장 하나를 1:1로 대응한다.

  실제 변경 내역을 알린다. 포털 프로필 저장은 실제 건수를, ClickUp·Discord·제안 발송은 0건을
  각각 구분한다. 내부 검사명이나 작업 장부 색깔을 그대로 말하지 않는다.
