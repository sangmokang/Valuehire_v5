## 작업 지시 (이대로 실행)

엄격 모드(SOT-30)로 다음을 수행하라. **새 스킬을 만들지 않는다** — SOT-30 확장 + 3진입점 반영 + 레포 러너/훅 구현이다(별도 스킬 = 드리프트 재발 경로).

### 핵심 설계 원칙 (사장님 확정 의도 — 약화 금지)

**"판단은 모델, 실행은 코드."** 프롬프트(말)는 강제가 아니다. 지시는 반드시 기계 수준으로 승격하며, 실행기별 강제 장치를 다음과 같이 구현한다(2026-07-19 사장님 확정):
- **Claude Code = 훅으로 강제.** PreToolUse 차단(러너 리스 마커 없는 직접 브라우저 타이핑 도구 호출 거부) + Stop 차단(스펙 내 사항 질문으로 끝나는 턴 반려, stop-evidence-gate 패턴 확장).
- **Codex = 훅이 없으므로 "유사 기능"을 직접 구현한다** (훅이 있는 척 금지 — 등가 장치 3종):
  1. **도구 내장 가드(Tool-embedded Guard)**: 강제 로직을 클라이언트가 아니라 **공유 도구 계층 자체에 심는다.** 포털 러너·CDP 유틸이 실행 시 러너 리스 마커(파일/env)를 스스로 검사하고 없으면 즉시 거부(exit 비0 + 사유 stderr). 어느 실행기가 부르든 도구가 자기를 지키므로 Codex에서도 동일하게 걸린다.
  2. **git 훅(pre-commit·pre-push)**: 커밋/푸시 시점 게이트(RED 원장·strict:gate·마커 검사)를 git 훅으로 실행 — git은 실행기 무관이라 Codex에서도 강제된다.
  3. **종료 등가 게이트**: Codex 세션은 Stop 훅이 없으므로, $st 스킬이 마지막 단계에서 `strict:gate` 류 종료 검사 러너를 의무 실행하고 그 exit 0 출력이 없으면 "완료" 선언 금지 + V1 검증 항목("불필요 질문 0건·종료 게이트 출력 존재")으로 이중 확인.
- 원칙: 강제 로직의 본체는 레포 코드(도구 가드·git 훅·게이트 러너)에 두고, Claude 훅은 같은 본체를 호출하는 추가 잠금이다 — 로직 이중 구현 금지(한 곳에 두고 양쪽에서 호출).

### 반영할 규칙 5개

**R1. 분해 → 스펙 → 예외 열거 (착수 의무 산출물, L2+ code-change)**
goal 문서에: ① 작업 분해표(단위 = AC 1개 = 검증 1개, 의존관계 포함) ② 단위별 계약(입출력 모양) ③ **예외 케이스 표** — 내가(모델) 먼저 예외를 산정해 열거(빈값·로그인만료·캡차·중복·셀렉터 실종·rate limit·사장님 크롬 점유 등)하고, 각 행에 "자동 처리 방법" 또는 "명시적 중단(fail-fast)+사유"를 확정. 마지막 행은 "그 외 전부 → 명시적 중단". **표에 없는 상황을 만나면 임의 판단 금지 — 중단하고 표를 갱신한 뒤 재개.** (기존 §1-11 입력 영역 표를 예외 케이스까지 확장하는 것.)

**R2. 질문 금지 게이트 (허들 제거)**
- 스펙·SOT·예외 표에 답이 있는 사항을 사용자에게 묻는 것 = 위반. 되묻지 않고 실행한다.
- 질문이 허용되는 유일한 경우: (a) 2FA·캡차·본인확인(SOT-26 INV) (b) 파괴적·비가역 행동 (c) 예외 표에 없는 신규 상황(이때도 "질문"이 아니라 "중단 보고+표 갱신안 제시").
- 기계 강제: Claude 쪽은 Stop 훅에 "턴이 스펙 내 사항 확인 질문으로 끝나면 차단·재진행" 검사를 추가(v4 stop-evidence-gate 패턴 재사용). Codex 쪽은 종료 등가 게이트(핵심 설계 원칙 ③)로 동일 효과 구현 — $st가 종료 검사 러너 실행을 의무화하고, 그 러너가 세션 산출물에서 미해결 질문·미커밋 잔존을 검사해 exit 비0이면 완료 선언 금지. V1 검증 항목에 "불필요 질문 0건" 체크 포함.

**R3. 위험 단계 러너 소유 + read-back 의무 (2026-07-19 재검증 반영 — 과장 금지)**
정직한 전제: 모델의 브라우저 접근을 **원천 봉쇄하는 것은 불가능**하다(Claude 훅은 도구 호출만 막고 Bash/CDP 우회가 존재, Codex는 훅 자체가 없음). "절대 못 한다"류 서술 금지. 대신 3단 방어:
- **역할 분담(정책+게이트)**: 항로 판단(페이지 상태 인식·예외 감지)은 모델, 손 조작(로그인·검색어 입력·발송·창 조작)은 레포 러너가 수행. 근거 — 정해진 칸에 정해진 값은 스크립트가 결정론적으로 우세하고 read-back 검증이 이미 구현돼 있음(`portal_worker.py` readback verification, 봇방지 페이싱 PC-E1). 단 셀렉터 실종·화면 개편 시 러너는 fail-fast하며, 이때 모델이 손으로 때우지 않고 셀렉터 사전(SOT-23)을 수리해 러너로 재시도한다.
- **보편 강제(핵심 — 두 실행기 공통)**: 누가 입력했든 **위험 입력(검색어·수신자·본문·발송 대상)은 전송/실행 전 read-back 대조 통과가 의무** — 넣은 값을 다시 읽어 계약과 대조, 불일치 시 진행 금지+기록. 안정성의 실체는 "누가 치느냐"가 아니라 "친 뒤 대조"다. (RPS Keywords 칸 오입력은 이 대조에서 걸린다.)
- **Claude 한정 추가 잠금**: PreToolUse 훅으로 러너 리스 마커 없는 직접 브라우저 타이핑 도구 호출을 차단(우회 가능함을 명시하되 주요 경로를 좁힌다). 로그인은 SOT-26 INV1대로 러너가 무조건 수행하고 성공 판정 전 창 닫기 금지(러너가 창 수명 소유). 러너 실패 시 같은 시도 기계 반복 금지 — 원인 기록 후 예외 표 절차.

**R4. 재발 원장 (반복 프롬프팅 = 사장님 의도 승격)**
- `docs/sot/`에 재발 원장 파일 신설: 사장님 교정 1건 = 1행(`날짜 | 지적 | 현재 강제수준 policy/runner/hook | 승격 기한`). 오늘 6건을 초기 데이터로 등록.
- **같은 지적 2회째 = 자동 승격 의무**: 해당 항목을 러너/훅으로 올리는 작업을 즉시 발행(기존 SOT-30 §0 "검토한다"→"의무"로 강화). 원장에 2회 이상인데 policy에 머문 항목이 있으면 strict:gate가 경고.
- 모든 strict 작업 착수 시 원장을 읽고 관련 항목을 goal에 인용(과거 회수의 일부).

**R5. 단위 관문 (v1 유지)**
단위 N 검증(RED→GREEN / live 실증 / 근거-반례) 통과 전 N+1 착수 금지. 무의존 단위만 병렬 워크트리 허용. 검증 방식은 단위마다 test/live/evidence 중 선언. 분해표 재분해 허용 + 사유 로그. 최종 V1/V2는 코드만이 아니라 분해표·예외 표 자체를 공격("표에 없는 현실 케이스 있는가").

### 추가 규칙 3개 (v3 — robustness-briefing 2026-07-19 2부에서 신규 채택)

**R6. 실패 = 평가 케이스 파이프라인 (Anthropic evals · OpenAI Evals)**
사장님 신고 실패 1건 = 재발 원장(R4) 1행 = **회귀 케이스 1건**. 수백 개가 아니라 실제 실패 사례에서 뽑은 20~50개로 시작(QA 이슈로그 348건이 원료), 안정되면 배포 전 100% 통과 회귀 스위트로 졸업. 원장 `상태` 열에 대응 케이스 경로를 기록.

**R7. 정정 2회 = 세션 리셋 (Claude Code Best Practices · OpenAI Building agents)**
같은 작업에서 정정을 2회 이상 받으면 그 대화를 계속 때우지 않는다 — 배운 것을 반영한 더 나은 프롬프트로 새로 시작하는 편이 빠르다. 리셋 전에 원장 + goal 문서에 배운 것을 기록(다음 세션의 입력).

**R8. 위임·보고 위생 (Anthropic multi-agent/context engineering · GPT-5.1 Prompting Guide · Google Bake-Off)**
① 하위 에이전트 위임 = 4칸 템플릿 고정(목표 / 반환 형식 / 쓸 도구 / 하지 말 것) + 실패 시 전체 재시작 대신 체크포인트 재개 ② **실제 돌리지 않은 타입체크·테스트는 언급 금지**(허위 완료 보고 차단) ③ 장시간 작업은 도구 호출 6~8회마다 진행 보고 ④ 지침·가드는 "이 줄을 지우면 실수하는가?"로 가지치기(규칙 과다 = 과잉 방어·실행 회피 재발 경로) ⑤ 복잡도는 단일 호출→단순 워크플로→에이전트 순서로만, 도구 인자는 실수 불가능하게(poka-yoke).

**이미 반영돼 중복 추가하지 않은 항목(2부 대조 결과)**: 합격/불합격 신호·검증 없으면 배포 금지·재현 실패 테스트 먼저(=SOT-30 core 3·4·6, RED-first) · "생각은 AI, 실행은 코드"(=R3) · 같은 오류 반복→규칙 승격(=R4) · 에러 메시지에 정식 경로 안내(기존 가드 패턴) · 스펙=대상 파일+비범위+검증 절차(goal 템플릿) · 외부 부작용=사람 승인(L3 owner signoff).
