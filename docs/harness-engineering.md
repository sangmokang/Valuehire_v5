# Harness Engineering — 코드 구현 원칙 (자동 로드 대상)

> 이 문서는 **"무엇을 믿고 어떻게 짜는가"의 원칙**이다.
> **"어떤 순서로 게이트를 통과하는가"의 절차**는 `docs/harness.md`에 있다. 둘은 한 쌍이다.
> 코드 작업이 감지되면 `.claude/hooks/harness-engineering-loader.sh`가 이 두 문서를 자동으로 상기시킨다.

---

## 0. 대전제 — 내 코드는 믿지 않는다 (SOT 불변식 5)

생성자(Claude)가 만든 코드는 **늘 헛점이 있다고 전제**한다. 칭찬하지 않고 깨뜨린다.
"됐다"는 **두 번 깨고도 안 깨졌을 때만** 성립한다.

```
작게 만든다 → 내가 먼저 깬다(패스1) → codex:rescue가 또 깬다(패스2) → 그제야 가져온다
```

시작 전엔 **과거 지시부터 회수**한다(중복 구현 금지).

---

## 1. 짜는 순서 (모든 코드 작업 공통)

| # | 단계 | 기준 |
|---|---|---|
| 0.5 | **과거 회수** | memory·코드·스킬·문서 최소 3축 grep. 있으면 재사용/확장. |
| 1 | **스펙** | 인수 기준 1개("무엇이 참이면 끝") — 기계 단언 우선. |
| 2 | **RED 먼저 (TDD)** | `make task NAME=...`로 worktree. 실패하는 테스트부터 커밋. |
| 3 | **작은 단위 구현** | 파일 1~5개, diff 50~300줄. 큰 덩어리 금지. RED→GREEN 최소 변경만. |
| 4a | **기계 검증** | `./verify.sh` exit 0. 출력 숫자 그대로 보고. |
| 4b | **2패스 적대검증** | 패스1 자기 반증 + 패스2 codex:rescue. 둘 다 못 깨야 통과. |
| 5 | **배송** | `make ship` → PR. CI 초록 + merge 전까지 "완료" 없음. |

> 큰 작업이면 인수 기준 1개로 못 적는다 → **쪼개서 별도 작업으로.** 작은 단위가 원칙이다.

## 2. 적대검증 체크리스트 (패스1 — 스스로 깬다)

코드를 사장님께 가져가기 전, 다음을 **일부러 넣어** 깨지는지 본다:

- 빈 값 / null / 빈 문자열 / 빈 리스트
- 잘못된 입력 / 타입 불일치 / 비정형 데이터
- 경계값 (0, 음수, 초과 길이, 최대치)
- 막힌 사이트: 401/403/404/429/5xx, captcha, 타임아웃
- 중복 (같은 URL·같은 후보·같은 포지션 2번)
- 동시성 / 재실행(resume) 안전성
- secret·토큰·개인정보 노출
- 실패를 "결과 없음"이 아니라 **원인별로** 구분(차단/쿼리부족/근거부족/스키마실패)

하나라도 깨지면 → 게이트 3으로 되돌아간다.

## 3. 적대검증 (패스2 — codex:rescue)

`codex:rescue` 스킬 / `codex:codex-rescue` 에이전트에 **같은 코드를 넘겨** 다른 눈으로 또 깬다.
Codex가 결함·반례를 찾으면 GREEN 아님 → 게이트 3. **이건 선택이 아니라 게이트다.**

## 4. 9개 하니스 (시스템성 작업일 때 — AI Search 등)

단발 스크립트가 아니라 "반복 실행·검증 가능·관측 가능한 시스템"을 만들 땐 아래를 함께 설계한다
(출처: `valuehire-ai-search-harness-engineering-prompt.md`).

1. **Input** — 입력 수집·정규화, 누락/중복/비정형 테스트
2. **Strategy** — 재현 가능한 전략 schema, 민감조건→역량조건 변환
3. **Discovery** — 채널별 adapter 분리, 차단/429/captcha 관측
4. **Evidence** — source URL 접근성 검증, 주장↔근거 연결
5. **Scoring** — 동일 rubric, 점수엔 근거·리스크·출처 필수, 낮은 근거는 점수 상한
6. **Privacy & Safety** — 공개 연락처만, 발송은 자동 금지(사람이 마지막에)
7. **Output** — 사람용 + 기계용(JSON), 스키마 검증 실패 시 미확정
8. **Evaluation** — golden set 회귀 테스트, 모델·프롬프트·adapter 변경 시 동일 벤치
9. **Observability** — run_id, 단계별 분리 로깅, 실패 원인 분류

## 5. 앤트로픽 *Building Effective Agents* 정렬

> 출처: anthropic.com/engineering/building-effective-agents — 공식 가이드의 원칙을 우리 하니스에 맞춘 것.

| 앤트로픽 원칙 | 우리 하니스에서의 적용 |
|---|---|
| **단순하게 (Simplicity)** | 작은 단위·최소 변경(게이트 3). 복잡도는 "결과가 분명히 좋아질 때만" 올린다. |
| **투명하게 (Transparency)** | 계획·게이트 통과 과정을 노출. verify 출력 숫자를 그대로 보고. |
| **도구 설계 (ACI)** | 스킬/툴은 문서·예시·경계·엣지케이스를 명시. 실수하기 어렵게(Poka-yoke). |
| **샌드박스 테스트** | worktree 격리 + `./verify.sh`. 프로덕션 전 충분히 깨본다. |
| **가드레일** | 발송 자동금지·사장님 chrome 양보(R4)·secret 차단. 누적 오류 차단. |
| **워크플로 vs 에이전트** | 정형 작업은 게이트 워크플로로, 모델 판단이 필요한 곳만 자율. |

## 한 줄 요약

> **작게 만들고 → 내가 깨고(패스1) → codex:rescue가 또 깨고(패스2) → verify 통과 → 그제야 "완료".
> 시작 전엔 과거 지시부터. 단순·투명·검증은 앤트로픽 공식 원칙과 같은 방향.**
