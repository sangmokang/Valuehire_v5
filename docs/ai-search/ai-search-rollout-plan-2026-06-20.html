맥에서 3사 로그인 백그라운드 상주화

부분 → 완성

지금. scripts/portal_browsers.sh start가 nohup로 3사 크롬을 띄워 터미널을 닫아도 살아있게 한다(프로필 재사용, 127.0.0.1 CDP 고정). 하지만 맥 재부팅·로그아웃·크롬 크래시 후 자동 복귀는 없다.

할 일 (이미 있는 스크립트를 서비스로 승격)

  • launchd 등록. 기존 scripts/launchd/com.valuehire.search-runner.plist 패턴을 본떠 com.valuehire.portal-browsers.plist를 만들고 portal_browsers.sh start를 RunAtLoad + KeepAlive로 등록 → 로그인 시 자동 켜짐, 죽으면 재시작.
  • 로그인 헬스체크. portal_browsers.sh status를 확장 — 각 포트의 현재 URL이 로그인 페이지(예: /auth, /login, LinkedIn checkpoint)면 "로그인 끊김"으로 표시하고 사장님께 한 줄 알림. 자동 재로그인·캡차 풀이는 하지 않는다(SOT4) — 사람이 그 창에서 한 번 로그인.
  • 양보 가드(SOT2). 사장님이 직접 크롬을 쓰는 신호가 보이면 그 머신의 자동 검색을 멈춘다. (포털 안정성 계층의 R4 양보 규칙과 연결.)
  • 로그·프로필 경로는 현재대로 ~/.valuehire/ 유지(데이터 보존, 싱글톤 락은 고정 허용목록만 정리 — 이미 GREEN #24).
완료 조건: 맥 재부팅 후 로그인하면 1~2분 내 3사 크롬이 자동으로 떠 있고 portal_browsers.sh status가 세 포트 모두 살아있음을 보고 · 세션을 닫아도 유지 · 로그인 끊김이 status에 정확히 표시 · 사장님 크롬 사용 시 자동작업 정지 확인.
P2

싱글 포지션 서치 완수 — 검색 필터 정교화

최약점

지금. 싱글 서치의 뼈대(search 스킬·스코어링·정규화)는 GREEN이지만, 실제 포털에서 검색을 거는 부분이 약하다 — 키워드 1개만 입력하고 연차·지역·업종 필터를 안 쓰며, 결과카드 셀렉터가 라이브 DOM과 어긋나 비후보(회원가입 링크 등)를 긁는다. 골 프롬프트 search-filter-precision-goal-prompt.md에 해야 할 일이 이미 정리돼 있다.

할 일 (골 프롬프트대로, 작은 기능 단위로)

  • JD → 검색 스펙. 직무(타이틀 동의어)·업종·연차·지역·타겟사·제외조건을 한국어+영어 둘 다로 구조화한 공유 스펙 파일.
  • 매 검색 전 초기화 후 재세팅(가장 중요). 화면에 남은 AND/OR/NOT 값을 전부 지우고 이 포지션 키워드로 새로 채운다 — 이전 검색 잔여 필터 오염 방지.
  • 채널별 facet 적용. 사람인(OR=직무·AND=업종·NOT=제외·연차 #career_min/max) / 잡코리아(#txtKeyword·지역·연차) / LinkedIn(Keywords Boolean 1순위 + YoE 2순위 + Location=South Korea). 연차는 JD ±1~2년 폭.
  • 결과카드 셀렉터 보강. 사람인·잡코리아 RESULT_CARD_SELECTORS를 라이브 DOM에 맞춰 실후보만. LinkedIn은 lazy-load 스크롤 후 a[href*="/talent/profile/"].
  • 패스 규칙. 이직 잦은 후보·Freelancer 표기는 최하위/제외, risk_or_gap 명시.
  • 이슈 #27 종결. search 스킬 line150 LinkedIn 기술 불일치 닫기.
완료 조건: 실제 포지션 1건으로 3사 각각 한·영 합집합 롱리스트 산출 + 연차·업종·직무 일치 후보 수 보고 · 비후보 오탐 제거 확인 · 외부 발송 0건 · 캡차 감지 시 정지 · ./verify.sh 통과. (내가 한 번, Codex Rescue가 한 번 — 두 번 적대검증 후 보고 / SOT5)
P3

자가학습 · 평가 루프

미착수

지금. 학습·평가의 부품은 이미 GREEN이다 — scoring.py(품질 4기준), ab_harness.py(블라인드 A/B·순도 리포트), reservoir_log.py(무조건 저장 큐), match.py(코사인 top-K 재랭킹). 빠진 건 "검색 결과를 다시 필터 설정에 물리는 고리"다.

할 일

  • 측정 정의. "더 나은 서치"의 숫자 정의 — 일치 후보 비율, 상위 점수대 후보 수, 사람이 "컨택"으로 올린 비율(승인 신호) 등을 채널·포지션별로 reservoir_log에 적재.
  • A/B로 필터 변형 비교. 같은 포지션에 키워드 세트 A vs B(예: 동의어 폭, 연차 ±1 vs ±2, Boolean 조합)를 ab_harness로 블라인드 비교 → 이긴 설정을 그 직무군 기본값으로.
  • 피드백 반영. 사람이 최종 선택한 후보(컨택/보류/제외)를 라벨로 모아 스코어링 가중치·키워드 매트릭스를 직무군별로 보정. 자동 발송은 절대 없음 — 학습 신호는 "사람이 누른 결과"에서만 가져온다.
  • 민감조건(나이·성별·학교서열)은 학습 피처로 쓰지 않는다 — 경력·역량·성과로 변환(스킬 SOT).
완료 조건: 한 직무군에서 A/B 2회 이상 돌려 "이긴 필터 설정"이 기록되고, 그 설정 적용 후 일치 후보 비율이 baseline 대비 개선됨을 숫자로 보임(순도 리포트로 후보 누설 차단 확인).
P4

맥북 + 맥에어 2대 분산

PoC 2단계 완료 → 실기기

지금. cdp-poc가 단일 맥에서 오케스트레이터·Redis·워커3·워커전용 크롬을 Phase2까지 GREEN으로 검증("3단계 2대 실기기 진행 가능"). 설계는 옵션B 확정 — 중앙은 작업만 분배, 각 머신 워커가 자기 127.0.0.1 CDP에만 붙는다.

할 일

  • 역할 배치. 맥북 = 오케스트레이터 + Redis + 워커1, 맥에어 = 워커2. 각 머신은 P1의 상주 크롬(자기 포털 로그인)을 자기 것만 제어.
  • Tailscale 메시 VPN. 두 맥을 같은 tailnet에. 오가는 건 작업 지시뿐 — Redis 접근은 Tailscale IP로, CDP 포트는 절대 tailnet/0.0.0.0에 노출하지 않는다(SOT5). Redis는 이때 인증/접근제어 필요(PoC는 단일노드·무인증이었음 — 2대부터 보강).
  • 작업 라우팅. "이 포지션은 이 머신의 사람인 세션에서"처럼 채널·로그인 보유 머신에 작업을 보내도록 큐 라우팅.
  • 워커 죽음·리스 만료·크롬 좀비 정리는 PoC에서 이미 검증된 로직 재사용(리스 펜싱·terminal idempotency·프로필 락 정리).
완료 조건: 맥에어를 꺼도 맥북 작업이 계속되고(반대도) · 맥에어 워커 강제 종료 시 리스 만료→재배정으로 작업 회수 · 양쪽 CDP 포트가 lsof상 127.0.0.1에만 LISTEN(tailnet 노출 0) · 2대 합산 부하 테스트 invariant 위반 0.
P5

사무실 윈도우 4대 합류 — 총 7대

미착수 (포팅 리스크 큼)

지금. 옵션B 설계는 OS와 무관하지만, PoC와 상주 스크립트는 macOS에서만 검증됐다(launchd·nohup·Chrome 경로·프로필 락 정리 모두 맥 전용). 윈도우는 별도 검증이 필요하다.

할 일 (한 대 먼저 검증 후 4대)

  • 상주 방식 포팅. launchd → 윈도우 작업 스케줄러 또는 NSSM 서비스. nohup 대체.
  • Chrome/CDP 포팅. Chrome 실행 경로·--remote-debugging-address=127.0.0.1 동작·프로필 싱글톤 락 정리(SingletonLock 등)를 윈도우에서 재검증.
  • Node/Redis 클라이언트. 워커(Node+Playwright+ioredis)를 윈도우에서 빌드·기동. Redis는 중앙(맥) 한 곳, 윈도우는 Tailscale로 큐만 접근.
