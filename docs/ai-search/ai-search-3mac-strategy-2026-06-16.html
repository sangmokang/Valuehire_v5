맥북 · 맥미니 · 맥에어 세 대로 후보 검색(AI Search)을 돌리는 최고의 전략입니다. 로그인 · 세션 유지 · 보안 · 링크드인 1기기 제한 · 멀티 서치 · admin 공유 · 디스코드 주기 · 이력서 저장 여덟 관점을 실제 코드를 읽고, Codex로 적대 검증해 정리했습니다.

한 줄 요약

"맥 3대"는 속도 3배가 아닙니다. 같은 회사 계정 하나를 세 대가 나눠 쓰는 거라, 한 번에 한 대만 그 계정으로 일해야 안 막힙니다.

그래서 먼저 만들 한 가지 → "지금 이 계정은 어느 맥이 쓴다"를 정해 주는 중앙 정리원(계정 코디네이터)입니다. 이게 없으면 3대는 그냥 한 계정을 두고 서로 싸우는 자동화 3개일 뿐이라, 사이트가 계정을 막아버릴 수 있습니다.

01 — 결론부터

기기별 역할 — 한눈에

세 대를 "속도 분담"이 아니라 "역할 분담 + 비상 교대"로 씁니다. 사람인·잡코리아·링크드인 계정은 한 세트뿐이니, 그 계정으로 동시에 일하는 맥은 항상 한 대입니다. (아래는 권장안 — 사장님이 평소 쓰는 노트북이 맥에어라면 맥북↔에어 역할만 맞바꾸면 됩니다.)

사람 검토·발송기

💻 MacBook

링크드인 RPS 전용상시자동 X

  • RPS를 "한 대"에 못 박는 잠금(아래 04)을 이 기기가 쥐는 동안 미니·에어는 RPS를 절대 안 켬.

  • 운영 허브: 세션 저장(스냅샷) 주인, 디스코드·admin 보고 단일 발신자, 이력서 저장 단일 기록자, 일일 한도 원장.

콜드 스탠바이 · 공개웹

💻 MacBook Air

평소 대기RPS 절대 X
  • 사람인·잡코리아 계정은 평소 꺼둠(콜드 스탠바이). 미니가 죽으면 그때만 잠금을 넘겨받아 교대 가동.
  • 공개 웹 검색 전담(ChatGPT·공개 프로필·GitHub·Scholar) — 보호 계정이 아니라 동시 실행해도 안전.

왜 이렇게? — 사이트(사람인·잡코리아·링크드인)는 "같은 계정이 두 곳에서 동시에 활동"하면 한쪽을 강제로 끊거나 계정을 막습니다. 그래서 보호 계정은 한 번에 한 대, 나머지는 대기·공개웹·사람발송으로 역할을 가릅니다.

02 — 가장 중요한 진실

"맥 3대"의 함정 — 계정은 한 세트뿐이다

맥이 3대라도 사람인 기업회원 1개, 잡코리아 서치펌 1개, 링크드인 RPS 1석입니다. 기기를 늘려도 계정이 늘지 않습니다. 그래서 진짜 위험은 "느림"이 아니라 "한 계정을 두 대가 동시에 만지는 것"입니다.

❌ 흔한 착각

"맥 3대니까 사람인을 미니·에어 두 대에서 같이 돌리면 2배 빠르겠네." → 사이트가 동시 접속을 감지해 한쪽을 강제 로그아웃 / 보안문자 / 계정 제한. 오히려 더 느려지고 계정이 위험해집니다.

✅ 맞는 그림

한 계정 = 한 번에 한 대. 평소 미니가 사람인·잡코리아를 전담, 에어는 대기. 미니가 멈추면 에어가 잠금을 넘겨받아 이어받음. 3대의 가치는 속도가 아니라 "안 끊김(이중화)"입니다.

지금 코드의 한계(정직하게) — 현재는 한 컴퓨터 안에서만 같은 작업이 겹치지 않게 잠급니다 (portal_worker.py:203 flock). 컴퓨터끼리는 막는 장치가 없습니다. 그래서 "한 계정 한 대"를 진짜로 강제하려면, 아래 04의 중앙 잠금(분산 리스)을 새로 만들어야 합니다.

03 — 8대 관점 심층 분석

요청하신 여덟 관점, 하나씩

각 항목 옆 배지: 됨 코드에 이미 있음 · 부분 한 대용은 되나 3대용은 보강 필요 · 없음 새로 만들어야 함. 근거는 파일:라인으로 표기.

① 로그인 — "이상 없다"는 사람인·잡코리아까지. 링크드인 자동로그인은 알맹이가 비어있음 부분

사람인·잡코리아는 아이디·비밀번호 자동 입력·제출이 코드에 실제로 있습니다(portal_autologin.py:25 AUTO_LOGIN_SELECTORS). 세션은 launch_persistent_context의 로그인 저장 폴더(~/.valuehire/portal_profiles)에 남아 재사용됩니다.

적대검증이 잡은 모순 — 링크드인 RPS는 세션이 없을 때 자동로그인을 호출은 하지만 (portal_login.py:495), 실제 자동로그인 표(portal_autologin.py:25)와 로그인 URL 함수 (portal_autologin.py:74)는 링크드인을 거부/예외 처리합니다. 테스트도 이 자리를 가짜(mock)로 메워 진짜로 로그인 칸이 채워지는지 증명하지 않습니다(test_portal_preflight_autologin.py:340). → "3사 자동로그인 절대 막지 않는다"(SOT 1)와 코드가 어긋남. 멀티맥 RPS를 켜기 전에 이것부터 확정해야 합니다.

할 일: 링크드인 실제 로그인 URL·셀렉터를 구현하고 가짜 아닌 회귀 테스트로 증명하거나, 정책을 "RPS는 기존 세션만 확인·수동 재로그인"으로 명문화해 코드와 일치시킨다. 둘 중 하나로 모순을 없앤다.

② 세션 유지 — 암호화 스냅샷은 튼튼. 단 "기기끼리 공유"가 보안과 충돌 부분

로그인 상태를 VHSS1 암호 봉투로 묶어 Supabase(validated_session_snapshots)에 저장하고, 깨지면 재주입으로 복구합니다. 풀려도 대부분 자동 회복(지수 백오프), 보안문자는 사람에게 넘김 — 한 대 기준으론 잘 됩니다.

3대로 가면 깨지는 지점 — 스냅샷 키가 사이트+worker_id+종류뿐이라 "어느 기기/계정 소유"라는 칸이 없습니다(session-state SQL:16). 또 저장이 마지막에 쓴 사람이 이김(last-writer-wins) 방식이라(SQL:149,189,260), 에어가 오래된 스냅샷을 나중에 덮어쓰면 전 기기가 그 불량 세션을 복원해 로그아웃 루프에 빠질 수 있습니다.

할 일: 스냅샷 쓰기는 "그 계정의 잠금을 쥔 단 한 대"만 허용. 저장에 펜싱 토큰·기기ID·캡처시각(단조 증가) 가드를 넣어 옛 스냅샷이 새 것을 못 덮게 한다.

③ 보안 — 머신별 Keychain 키 모델 ↔ 스냅샷 공유가 정면 충돌 설계 필요

스냅샷 암호 키는 그 맥의 Keychain에서만 만들어지고 읽힙니다 (portal_snapshot.py:60), 없으면 기기마다 무작위로 새로 생성(:77), 복호화도 같은 키로 인증(:165). 비밀을 화면·로그에 절대 안 찍는 설계는 좋습니다.

딜레마 — 미니가 만든 스냅샷을 에어가 풀려면 같은 키가 있어야 합니다. 그런데 ① 키를 안 나누면 → 에어는 복호화 실패(공유 불가). ② 키를 3대에 복사하면 → 노트북 한 대만 잃어버려도 모든 포털 쿠키가 통째로 노출됩니다. 게다가 Supabase 스냅샷을 쓰려면 service-role 키가 필요한데, 문서는 이 키를 서버에만 두라고 못 박습니다(docs/search-access.md:12) — 노트북에 두면 안 됨.

할 일: 원시 복호화 키를 기기끼리 나누지 말 것. 중앙 브로커/KMS가 키를 쥐거나, 기기별 봉투암호화(각 맥은 자기 앞으로 다시 암호화된 스냅샷만 풀 수 있음)로 간다. service-role 키는 미니(허브)에만, 노트북엔 두지 않는다.

④ 링크드인 RPS 1기기·1브라우저 — 진짜로 막으려면 "펜싱 리스"가 필요 없음(핵심)

현재는 한 컴퓨터 안에서 worker_id=default 단일 프로필만 허용하고 OS 파일락으로 직렬화합니다 (portal_worker.py:172,214). 이건 "한 컴퓨터 안"에서만 유효합니다 — 다른 맥은 못 막습니다.

split-brain(둘 다 주인이라 착각) 시나리오 — 단순 TTL 잠금만 두면: 맥북 하트비트가 잠깐 멈춤 → 잠금 시간만료 → 미니가 잠금 획득 → 그런데 맥북의 옛 크롬 자동화가 아직 살아있음 → 두 대가 한 링크드인 계정을 동시에 건드림 → 계정 위험. 시간만료(TTL)만으론 부족합니다.

할 일: DB 서버시간 기반 compare-and-set(원자적 잠금) + 펜싱 토큰을 만든다. RPS 행동·스냅샷 저장 직전마다 토큰을 다시 확인하고, 하트비트가 끊기면 행동 전에 먼저 멈춘다(stop-before-action). "딱 한 기기·한 브라우저"는 이 잠금으로 비로소 진짜가 됩니다.

⑤ 멀티 서치 — 큐·중복제거·점수화는 됨. 단 일일 한도가 기기별이라 3배 과소진 위험 부분

여러 포지션을 묶어 채널별로 검색·중복제거·역매칭·점수화하는 큐는 구현돼 있습니다 (queue_runner.py, multisearch SKILL). 멈춰도 pending으로 두고 이어받기(resume) 안전.

3대로 가면 — 하루 호출 한도(예: 사람인 API 500/일)를 러너마다 따로 세는 구조라 (portal_runtime.py:83, portal_ops.py:345, portal_queue_executor.py:64), 3대가 각자 "오늘 0회"로 시작해 합쳐서 한도의 3배를 써버릴 수 있습니다. 또 크롬 연결이 끊기면 사람인·잡코리아까지 전부 같이 멈춥니다(queue_runner.py:69) — 채널이 묶여 있음.

할 일: 중앙 한도 원장(모든 맥이 한 숫자를 본다)을 둔다. 크롬 연결 상태는 채널별로 분리해 링크드인이 끊겨도 사람인·잡코리아는 계속 돌게 한다.

⑥ admin.valuehire.cc 공유 — 현재 연동이 아예 없음 + PII 게이트 필요 없음

지금 결과 공유처는 ClickUp Activity 코멘트 + 디스코드뿐입니다. admin.valuehire.cc로 보내는 코드는 저장소 어디에도 없습니다(grep 무결과). 새 레일을 만들어야 합니다.

주의(개인정보) — admin 웹앱으로 후보 정보를 밀어넣는 건 새로운 개인정보 노출면입니다. 누가 쓸 수 있는지(쓰기 권한·RLS)가 불분명하면 위험합니다. SOT/운영원칙상 공개·동의된 연락처만 다뤄야 하므로(multi-position 문서:86), 이력서 원문·OCR 본문을 admin으로 그대로 푸시하면 안 됩니다.

할 일: "미니(허브)만 admin에 쓴다 → admin은 읽기"로 단방향. 행 단위 권한(RLS)·서버 키로 보호. 푸시 내용은 요약·점수·근거 URL·공개 프로필 링크까지만. 원문·연락처 원본·OCR 덤프는 제외.

⑦ 디스코드 일관 주기 — 알림은 되나 "정기 발송"은 약함. 3대가 각자 보내면 중복/스팸 부분

재인증 알림과 후보 브리핑 포맷(format_discord_candidate_briefing)은 있습니다(discord_briefing.py:12). 하지만 "정해진 주기로 결과를 한 번씩 보내는 스케줄러"는 약하고, 주로 사건 알림 위주입니다.

3대 문제 — 세 대가 각자 브리핑을 쏘면 같은 내용이 3번 옵니다(스팸). 미니가 죽으면 아무도 안 보내는 공백도 생깁니다.

할 일: 디스코드·admin 발송은 미니(허브) 단 한 대만. "발송 권한 잠금" 1개를 두고, 시간창별 멱등키(같은 시간대엔 한 번만)로 중복을 막는다. 미니 다운 시에만 에어가 발송권을 넘겨받음.

⑧ 이력서 저장(프로필 아카이버) — v5에 실물이 없음. 3대가 한 SQLite에 쓰면 충돌 없음

이력서/프로필을 화면·본문·OCR로 저장하는 프로필 아카이버는 v5 체크아웃에 없습니다 (어댑터 약속만 있음 — multi-position 문서:24,163). 실물은 v4(../Valuehire_v4/tools/profile-archiver)에 있습니다.

주의 — v4의 로컬 SQLite를 그대로 살려 세 대가 한 파일에 같이 쓰면 충돌·데이터 손상이 납니다. 게다가 현재 수확기는 첫 저장 오류가 나면 남은 후보를 버립니다(harvest_runner.py:142) — 조용한 누락.

할 일: 아카이버를 단일 서비스(미니에서만 기록) + 멱등키(같은 후보 중복저장 방지)로 이식. 여러 맥이 한 SQLite 파일을 공유하지 말 것. 저장 실패는 건너뛰되 남은 후보는 계속 저장하도록 고친다.

04 — 적대적 검증 결과

Codex가 코드를 읽고 깬 위험 목록

[ 중앙 계정 코디네이터 · Mac mini ] ├─ 펜싱 리스 "지금 saramin/jobkorea/linkedin_rps 는 어느 맥?" 06 — 실행 로드맵

작게 잘라, 순서대로 (하니스 게이트)

큰 덩어리 금지. 각 단계는 실패 테스트 먼저(RED) → 최소 구현 → 두 번 깨기 → 검증 → 올리기. 아래 순서는 "위험 큰 것부터, 의존 순서대로"입니다.

0

모순부터 없앤다 — 링크드인 자동로그인 (P0-5)

SOT 1과 코드가 어긋난 상태. 실제 RPS 로그인을 구현+실증하거나, 정책을 "세션확인·수동 재로그인"으로 정정. 이걸 정하기 전엔 멀티맥 RPS 시작 금지.

1

중앙 펜싱 리스 (P0-1, P0-4)

Supabase에 계정별 잠금 행(서버시간 CAS + 펜싱 토큰 + 하트비트). "한 계정 한 대"를 진짜로 강제. 행동 직전 토큰 확인·실패 시 정지.

2

한도 원장 + 채널 분리 (P1-8, P1-9)

일일 한도를 중앙 한 숫자로. 크롬 연결 상태를 채널별로 떼어 링크드인 끊김이 사람인·잡코리아를 멈추지 않게.

3

스냅샷 소유권·키 분리 (P0-2, P1-7)

리스 보유자만 스냅샷 쓰기 + 펜싱/단조 가드. 키는 중앙 브로커 또는 기기별 봉투암호화. service-role 키는 미니에만.

4

단일 발신자 — 디스코드·admin (P2-11, P1-10)

미니만 발송 + 시간창 멱등키. admin은 단방향·RLS·요약/근거만(원문·OCR 금지). 정기 주기 스케줄러 추가.

5

이력서 아카이버 이식 (P0-6)

v4 아카이버를 단일 서비스+멱등키로 미니에 이식. 한 SQLite 공유 금지. 저장 오류여도 남은 후보 계속 저장.

6

에어 페일오버 리허설

미니 강제 종료 → 에어가 리스 인수 → 옛 세션 충돌 없는지 확인. split-brain이 안 나는지 실측.

지금 당장 하면 안 되는 것 — 중앙 리스가 생기기 전에 사람인·잡코리아를 미니·에어 두 대에서 동시에 켜는 것, 그리고 링크드인 RPS를 두 대에서 켜는 것. 계정이 막히면 복구가 가장 비쌉니다.

07 — 점검표

매일 / 설정 한 번

처음 한 번

  • 역할 고정: 맥북=링크드인 발송 / 미니=사람인·잡코리아+허브 / 에어=대기+공개웹
  • 링크드인 RPS는 맥북의 전용 크롬 프로필 1개에만. 미니·에어엔 RPS 프로필 자체를 안 만듦
  • service-role 키·포털 비밀번호는 미니에만. 노트북엔 두지 않음
  • 중앙 펜싱 리스가 켜져 있고, 세 맥이 모두 그걸 보고 움직이는지 확인

매일

  • 같은 계정을 두 대가 동시에 켜지 않았는가 (특히 사람인·잡코리아)
  • 일일 한도를 합쳐서 70% 넘기지 않았는가 (중앙 원장 확인)
  • 보내기는 사람이 직접 눌렀는가 (자동 발송 0건)
  • 사장님이 맥북 크롬 쓰기 시작하면 그 맥 자동작업이 멈췄는가 (R4)
  • 디스코드 브리핑이 한 번만 왔는가 (중복 아님)
