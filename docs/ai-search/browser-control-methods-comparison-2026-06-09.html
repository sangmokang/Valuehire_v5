0. 한눈에 보기 (TL;DR)

"셸/코드가 브라우저를 제어한다"는 것은 결국 자동화 프로세스(Python·Playwright)가 어느 브라우저에 어떻게 붙느냐입니다. 크게 두 갈래입니다.

갈래한 줄 설명본 저장소 사용처
붙기 (Attach)사람이 이미 켜둔 브라우저에 원격 접속해 조종LinkedIn RPS
띄우기 (Launch)자동화가 자기 전용 브라우저를 새로 실행사람인 · 잡코리아
우리 코드는 채널별로 의도적으로 나눠 씁니다 (portal_worker.py:350~365). LinkedIn은 "붙기", 사람인·잡코리아는 "띄우기".

1. 네 가지 제어 방식

A connect_over_cdp — 켜져 있는 Chrome에 CDP로 붙기 ★ LinkedIn

# portal_worker.py:351
browser = await playwright.chromium.connect_over_cdp("http://127.0.0.1:9222")
context = browser.contexts[0]   # 사람이 쓰던 그 세션을 그대로 물려받음

CDP = Chrome DevTools Protocol. Chrome을 --remote-debugging-port=9222로 켜면 열리는 "원격 조종용 문". 자동화가 브라우저를 새로 띄우지 않고, 사람이 로그인까지 끝낸 실제 Chrome 창에 그대로 접속합니다. 공식 제약: Chromium 계열에서만 동작(Firefox/WebKit 불가).

B launch_persistent_context — 전용 Chromium을 프로필 폴더로 띄우기 ★ 사람인·잡코리아

# portal_worker.py:357
context = await playwright.chromium.launch_persistent_context(
    user_data_dir=profile_dir,   # 쿠키·로그인 상태가 디스크에 저장됨
    headless=False, args=[...],
)

자동화가 자기 전용 Chromium 창을 새로 실행. 사람 Chrome과 완전 별개. user_data_dir에 쿠키·localStorage가 저장돼 다음 실행 때 로그인 유지. 공식 제약 2가지: ① 같은 프로필 폴더로 두 개 동시 실행 불가, ② Chrome 기본 "User Data" 폴더 지정 금지(정책상 로딩 실패) → 반드시 별도 빈 폴더. 우리가 ~/.valuehire/portal_profiles/<채널>/<worker>로 분리하는 이유.

C. launch + new_context

매번 로그인·쿠키 없는 깨끗한 상태. 테스트·CI에 이상적. 우리 포털엔 부적합(매번 재로그인).

D. connect (Playwright 서버)

CDP가 아닌 Playwright 전용 프로토콜. launchServer 서버에 접속, 양쪽 버전 일치 필요. 원격/도커 분리용. 우리는 미사용.

3. 왜 채널마다 다른 방식을 쓰나

LinkedIn → "붙기(CDP)"

LinkedIn은 봇 탐지·체크포인트가 매우 공격적. 자동화가 띄운 깨끗한 브라우저는 쉽게 막힙니다. 사람이 직접 로그인·2FA를 끝낸 진짜 세션에 올라타야 탐지 위험이 가장 낮음. 그래서 코드도 단일 headed 워커로 강제(portal_worker.py:170~174)하고 발송은 사람 게이트로 둡니다.

사람인·잡코리아 → "띄우기(Persistent)"

LinkedIn만큼 탐지가 빡세지 않아 전용 Chromium으로 충분히 안정적. 프로필 폴더에 로그인이 저장돼 한 번 로그인하면 세션 유지. 사람 Chrome과 완전 분리되어 사장님이 낮에 Chrome을 써도 새벽 자동화와 안 부딪힘.

4. 이번 변경(2026-06-09)이 바꾼 것 / 안 바꾼 것

변경 전변경 후

핵심: 이번 변경은 "붙기(A)" 방식의 접속 주소를 설정으로 뺀 것뿐. "띄우기(B)"를 쓰는 사람인·잡코리아는 단 한 줄도 영향받지 않습니다. (헬퍼: resolve_chrome_cdp_endpoint() @ portal_worker.py:31)

5. Q① "사람인·잡코리아도 CDP로 돌리면 더 좋은가?"

무조건 더 좋은 게 아닙니다. 운영 목적에 따라 갈립니다.

CDP로 바꿨을 때 ✅ 얻는 것❌ 잃는 것
사장님 로그인 Chrome 세션 재사용(재로그인 불필요) · 봇 탐지 회피 ↑ · 캡차/2FA 즉시 처리 백그라운드 무인 실행 불가(사장님이 그 Chrome 쓰면 충돌) · 동시 실행 어려움 · 사장님 Chrome 점유
우리 운영 목표는 "사장님 주무시는 새벽 03~05시 KST 무인 Cron 실행"입니다. 이 목표엔 Persistent(B)가 정확히 맞습니다 — 사장님 Chrome과 분리되어 간섭이 없고, 프로필+스냅샷으로 세션이 유지됩니다.
권장: 현 분리 유지. 사람인·잡코리아를 CDP로 바꾸는 건 "무인 새벽 자동화"를 포기하고 "사장님이 지켜보는 수동 실행"으로 전환할 때만 의미가 있습니다.

6. Q② Headless냐 Headed냐?

# portal_live_check.py:1381 등
mode = "headed" if channel == "linkedin_rps" else ("headless" if config.headless else "headed")
채널기본 모드강제 여부
LinkedIn RPS항상 Headed강제 — headed Chrome에 CDP로 붙어 headless 불가 (portal_worker.py:173)
사람인·잡코리아기본 Headed, --headless로 전환선택

결정적 차이 — "사람 개입(캡차·2FA)" 가능 여부

Headed (화면 보임)

캡차·2FA·체크포인트가 뜨면 사람이 풀 때까지 멈췄다가, 세션 재검증 후 자동 재개.

Headless (백그라운드)

사람 개입이 비활성화 → 캡차/체크포인트가 뜨면 그 채널은 그냥 not_ready로 멈춤.

상황권장이유
최초 로그인 / 자격증명 갱신 / 탐지 잦은 사이트Headed캡차·2FA를 사람이 풀어야 함
세션 유효한 정기 무인 실행(새벽 Cron)Headless 가능백그라운드·화면 불필요. 단 도전 발생 시 복구 못 하고 스킵
LinkedIn무조건 Headed선택지 없음
핵심 트레이드오프: Headless = 무인·백그라운드(but 캡차 만나면 포기) ↔ Headed = 복구 가능(but 화면·사람 필요). 무인 서버에서 Headless를 쓰려면 암호화 스냅샷 폴백(아래 Q③)이 세션을 받쳐줘야 안전합니다.

7. Q③ 우리 프로젝트의 "세션 문제" — 구조와 진실

7-1. 세션은 어디에 저장되나 (3중 구조)

층채널저장 위치역할
1차: 프로필 폴더사람인·잡코리아~/.valuehire/portal_profiles/...쿠키·로그인 영속. storage_state는 런치옵션으로 안 넘김 — 폴더가 곧 세션
1차: 실제 ChromeLinkedIn사장님 Chrome(CDP)사람이 로그인한 세션 그대로
2차: 암호화 스냅샷전 채널Supabase(암호화 바이트)1차가 깨지면 재주입(reinject)으로 복구 (portal_snapshot.py)

7-2. 세션이 끊기는 진짜 원인

① 포털 측 만료 — 일정 시간 후 세션 무효화  ·  ② IP/보안 경고·체크포인트  ·  ③ 프로필 손상(reauth_cause=profile_corrupt)  ·  ④ 동시 점유 충돌(같은 프로필을 둘이 열거나, MCP·Playwright가 같은 Chrome을 동시에 잡음)

7-3. 끊겼을 때 복구 흐름

세션 끊김 감지 (reauth_cause: http_401/403 · login_redirect · profile_corrupt)
│
├─ 사람인·잡코리아 → 키체인/.env 자격증명으로 자동 재로그인 시도
│                    실패 시 → 암호화 스냅샷 재주입 → 그래도 실패 시 not_ready
│
└─ LinkedIn → 자동 발송/조작 금지. Discord로 "재인증 필요" 알림 → 사람이 처리

캡차·2FA·체크포인트는 어떤 경우에도 우회하지 않습니다(SOT 불변식). 감지되면 멈추고 사람에게 넘깁니다.

⚠️ 가장 중요한 오해 — "세션 유지하려고 계속 열어두면 안 되나?"
안 됩니다. 런북이 명시적으로 금지합니다. "10분마다 프로필 열기" 같은 keepalive 하트비트는 포털이 봇·어뷰징으로 보고 계정을 더 빨리 막습니다. 허용되는 건 ① 검색 직전 on-demand readiness 체크, ② 로그인 만료 감지, ③ 수동 재로그인 알림 뿐.

즉 "세션이 끊긴다"는 건 버그가 아니라 설계상 정상입니다. 끊김을 억지로 막는 게 아니라, 빠르게 감지하고 안전하게 복구(자동 재로그인/스냅샷/알림)하는 게 우리 전략입니다.

8. Q④ "Claude in Chrome · Playwright… 접속 장치 혼돈" 완전 정리

혼란스러운 이유는 — 우리 프로젝트가 브라우저를 건드리는 경로가 실제로 3가지이기 때문입니다. 스크린샷의 "Claude(MCP)가 이 브라우저에 디버깅 시작" 배너도 이 중 하나.

#장치무엇을 조종어떻게사용처
1MCP claude-in-chrome사장님 실제 Chrome확장이 직접 클릭·입력(배너)대화형 스킬(talent-search 등)
2Playwright CDP사장님 실제 Chrome(포트 9222)원격 접속LinkedIn 포털
3Playwright Persistent별도 전용 Chromium새 창 띄움사람인·잡코리아
핵심 충돌 지점: 1번(MCP)과 2번(Playwright CDP)은 둘 다 사장님의 실제 Chrome을 노립니다. 동시에 같은 Chrome을 제어하려 하면 서로 간섭합니다. 3번(사람인·잡코리아)만 완전 독립이라 충돌 무관.

혼돈을 없애는 한 장 그림

🖥️ 사장님 실제 Chrome (포트 9222로 열림) ├── [장치 1] MCP claude-in-chrome 확장 ─┐ 같은 Chrome을 노림 └── [장치 2] Playwright CDP (LinkedIn) ─┘ → 동시 사용 금지!
🤖 별도 전용 Chromium (자동화가 띄움) └── [장치 3] Playwright launch_persistent (사람인·잡코리아) → 독립, 안전

충돌 방지 규칙

① MCP(1)와 Playwright-CDP(2)를 동시에 돌리지 말 것. LinkedIn 자동화 중에는 MCP로 같은 Chrome을 조작하지 않기.

② 사장님 Chrome 점유 가드 — 사장님이 Chrome 쓰는 중이면 CDP 자동화(LinkedIn)는 멈춤. 사람인·잡코리아(3)는 별창이라 계속 가능.

③ 포트 일원화 — CDP 접속 주소는 이제 VALUEHIRE_PORTAL_CHROME_CDP_ENDPOINT 한 곳에서만 관리. MCP와 Playwright가 같은 포트를 가정하므로 포트를 바꾸면 양쪽 다 맞춰야 함.
