# Search Skill 동시 주입/동시 호출 동작 논리

작성일: 2026-06-10
대상: Valuehire Hermes, Discord, 여러 쉘 창, Search Skill, Multisearch

## 한 줄 결론

Search Skill을 여러 쉘 창이나 Discord 메시지에서 동시에 호출해도 “스킬 파일 자체”가 섞이는 것은 아닙니다. 다만 같은 세션에서 동시에 들어온 요청은 Hermes Gateway의 세션 실행 가드에 의해 큐잉·인터럽트·스티어 중 하나로 처리되고, 서로 다른 세션/쉘 프로세스에서는 독립 실행처럼 병렬로 돌 수 있습니다.

비유하면 Search Skill은 레시피 파일이고, 각 Hermes 실행은 주방입니다. 여러 주방이 같은 레시피를 동시에 읽을 수는 있지만, 같은 냄비에 동시에 재료를 넣으면 순서 정리 규칙이 필요합니다. Discord의 같은 대화방/같은 세션이 그 “같은 냄비”입니다.

## 용어 정리

- Skill 주입: `skills/search/SKILL.md` 같은 절차 문서를 에이전트 프롬프트에 넣어, 이번 실행에서 따라야 할 업무 규칙으로 쓰는 것.
- 세션: Hermes가 대화 맥락을 묶는 단위입니다. Discord에서는 플랫폼·채널·스레드·사용자 설정 조합으로 세션 키가 만들어집니다.
- Gateway: Discord/Telegram 같은 외부 메시지를 받아 Hermes 에이전트 실행으로 연결하는 중간 서버입니다.
- 큐잉(queueing): 지금 실행 중인 작업이 끝난 뒤 다음 차례로 처리하도록 쌓아두는 방식입니다.
- 인터럽트(interrupt): 현재 작업을 중단시키고 새 메시지를 우선 처리하는 방식입니다.
- 스티어(steer): 현재 작업을 끊지 않고 다음 도구 호출 뒤에 새 지시를 끼워 넣는 방식입니다.
- Multisearch: 여러 포지션/여러 후보 서치 요청을 하나의 의도된 배치 작업으로 다루는 운영 스킬입니다. 임의 동시 입력과 다르게 “처음부터 묶어서 처리”하는 방식입니다.

## 1. 여러 쉘 창에서 동시에 `search`를 주입하는 경우

각 쉘 창에서 Hermes를 별도 프로세스로 실행하면 기본적으로 각각의 에이전트가 독립된 실행 컨텍스트를 가집니다.

동작은 다음과 같습니다.

1. 각 프로세스가 Search Skill 파일을 읽습니다.
2. 각 프로세스가 자기 대화 컨텍스트에 Skill 내용을 넣습니다.
3. 모델 호출, 도구 호출, 로그, 결과 생성은 각 프로세스에서 따로 진행됩니다.
4. Skill 파일은 읽기 전용 절차 문서이므로, 단순 주입만으로 파일 내용이 서로 합쳐지거나 오염되지는 않습니다.

주의할 점은 “공유 자원”입니다.

- 같은 결과 파일을 동시에 쓰면 마지막 저장이 앞선 저장을 덮을 수 있습니다.
- 같은 브라우저 프로필/포털 로그인 세션을 동시에 잡으면 클릭·검색 상태가 충돌할 수 있습니다.
- 같은 외부 API(ClickUp, Discord, 포털 등)를 동시에 때리면 속도 제한 또는 중복 저장이 생길 수 있습니다.
- 같은 후보를 같은 포지션에 중복 저장하는 문제가 생길 수 있습니다.

따라서 쉘 창 여러 개로 병렬 실행하는 것은 “프롬프트/스킬 관점”에서는 안전하지만, “외부 자원/쓰기 작업” 관점에서는 잠금·작업 ID·결과 파일 분리·중복 제거가 필요합니다.

## 2. Discord에서 같은 방에 여러 번 입력하는 경우

Discord Gateway 경로에서는 Hermes가 세션 단위로 실행 중인 에이전트를 추적합니다. 코드상 `gateway/run.py`에는 다음 보호 장치가 있습니다.

- 실행 시작 직후 `_AGENT_PENDING_SENTINEL`을 `_running_agents`에 넣어, 아주 짧은 비동기 틈 사이에 두 번째 메시지가 같은 세션으로 동시에 들어와도 “이미 실행 중”임을 알 수 있게 합니다.
- 같은 세션에 실행 중인 에이전트가 있으면, 새 메시지를 그냥 새 에이전트로 동시에 띄우지 않고 busy-input 정책에 따라 처리합니다.
- `/queue` 명령은 FIFO(먼저 들어온 순서)로 별도 차례를 만들며, 메시지를 합치지 않습니다.
- 일반 텍스트 follow-up은 설정에 따라 큐잉, 인터럽트, 스티어가 됩니다.
- 실행 중인 에이전트가 하위 에이전트(subagent)를 돌리고 있으면, 인터럽트가 하위 작업을 날려버릴 수 있어서 큐잉으로 낮춰 보호하는 로직이 있습니다.

즉 Discord 같은 세션에서 “서치해줘”를 여러 번 연속 입력하면, 보통 다음 셋 중 하나입니다.

1. queue 모드: 현재 Search가 끝난 뒤 다음 Search로 처리됩니다.
2. interrupt 모드: 현재 Search를 중단시키고 새 요청을 우선 처리할 수 있습니다.
3. steer 모드: 현재 Search 흐름에 새 지시가 중간 주입됩니다.

이 차이는 `display.busy_input_mode`, `display.busy_text_mode`, `/queue`, `/steer`, `/stop` 사용 여부에 따라 갈립니다.

## 3. 서로 다른 Discord 채널/스레드/DM이면?

세션 키가 다르면 별도 실행으로 볼 수 있습니다. 예를 들어:

- 사장님 DM의 Search 요청
- 다른 Discord 채널의 Search 요청
- 같은 서버 안의 다른 스레드 요청

이들이 서로 다른 세션 키로 잡히면 Gateway의 `_running_agents`도 세션별로 따로 잡히므로 병렬 실행될 수 있습니다.

다만 중앙 Hermes Gateway 하나가 처리하므로 CPU, 모델 API 속도 제한, 포털 자동화 세션, 외부 API 한도는 공유될 수 있습니다. 그래서 실행은 병렬이어도 병목은 한 곳에서 생길 수 있습니다.

## 4. Search Skill 자체는 상태를 가지나?

아닙니다. `skills/search/SKILL.md`는 절차 문서입니다. 이 문서를 여러 번 읽는 것 자체는 상태 변경이 아닙니다.

상태가 생기는 곳은 다음입니다.

- 대화 세션 기록
- 후보 결과 파일
- ClickUp/칸반/DB 저장 결과
- 브라우저 로그인 세션
- 포털 검색 페이지 상태
- Discord 메시지/스레드
- 모델·도구 실행 중 메모리

따라서 사고 포인트는 “스킬 충돌”이 아니라 “같은 포지션·같은 외부 저장소·같은 브라우저를 여러 실행이 동시에 만지는 충돌”입니다.

## 5. Multisearch와 임의 동시 입력의 차이

Multisearch는 처음부터 여러 검색 대상을 하나의 묶음으로 다루는 작업 방식입니다.

장점:

- 포지션별 우선순위를 한 번에 정할 수 있습니다.
- 후보 중복을 전체 관점에서 제거하기 쉽습니다.
- 결과 포맷을 통일하기 쉽습니다.
- 실행 순서와 저장 위치를 통제하기 쉽습니다.

반면 Discord에서 여러 번 입력하는 임의 동시 호출은 운영자가 보기에는 편하지만, 시스템 입장에서는 독립 요청 여러 개가 거의 동시에 들어오는 것입니다.

위험:

- 같은 포지션을 중복 서치할 수 있습니다.
- 후보 저장이 중복될 수 있습니다.
- 한 요청이 다른 요청을 인터럽트할 수 있습니다.
- 같은 브라우저 세션을 두 작업이 동시에 잡으면 검색 조건이 꼬일 수 있습니다.

## 6. Valuehire AI Search 관점의 권장 운영 원칙

### 안전한 경우

- 서로 다른 포지션을 읽기 전용으로 조사만 하는 경우
- 결과 파일 경로가 실행별로 분리된 경우
- 외부 저장/제안 발송 없이 dry-run으로만 도는 경우
- 각 실행이 서로 다른 세션 키/작업 ID를 가지는 경우

### 조심해야 하는 경우

- 같은 포지션을 여러 번 동시에 Search하는 경우
- ClickUp/칸반/DB에 후보를 저장하는 경우
- 사람인/잡코리아/LinkedIn 같은 포털 브라우저 세션을 쓰는 경우
- 제안 발송, 이메일 발송, 상태 변경 같은 되돌리기 어려운 작업이 포함된 경우

### 권장 방식

1. 단일 포지션 여러 개를 한 번에 돌릴 때는 Multisearch를 우선 사용합니다.
2. Discord에서 추가 요청을 넣고 싶으면 `/queue <요청>`을 사용해 순서를 명시합니다.
3. 진행 중인 Search에 조건만 보태고 싶으면 `/steer <조건>`을 사용합니다.
4. 잘못 돌기 시작한 작업은 `/stop`으로 끊고 새로 요청합니다.
5. 저장/발송 같은 live 작업은 포지션별 `run_id` 또는 작업 ID를 붙여 중복 저장을 막습니다.
6. 후보 저장 전에는 `(position_id, candidate_profile_url)` 기준으로 중복 확인을 합니다.

## 7. 권장 처리 정책표

| 상황 | 예상 동작 | 권장 |
|---|---|---|
| 여러 쉘 창에서 Search Skill 동시 주입 | 각 프로세스 독립 실행 | 결과 파일/브라우저/저장소 분리 |
| Discord 같은 DM에 연속 입력 | 같은 세션 busy 정책 적용 | `/queue` 또는 `/steer` 명시 |
| Discord 다른 채널/스레드 입력 | 세션이 다르면 병렬 가능 | 포털/저장소 충돌 주의 |
| Multisearch 사용 | 하나의 배치로 통제 | 여러 포지션이면 가장 권장 |
| live 저장/발송 포함 | 중복·충돌 가능 | owner 승인, 중복키, dry-run 선행 |

## 8. 구현 관점에서 필요한 가드

Search 파이프라인에는 다음 가드가 있으면 좋습니다.

1. `search_run_id`: 실행마다 고유 ID를 붙입니다.
2. `position_lock`: 같은 포지션에 대해서는 한 번에 하나만 저장 작업을 허용합니다.
3. `portal_lock`: 같은 브라우저/포털 세션은 한 번에 하나의 작업만 사용합니다.
4. `candidate_dedupe_key`: 후보 URL, LinkedIn URL, 이메일, 이름+회사 조합 등으로 중복을 막습니다.
5. `dry_run_default`: 기본은 읽기/요약만 하고, 저장·발송은 승인 뒤 실행합니다.
6. `result_namespace`: 결과 파일 경로에 날짜·포지션·run_id를 넣습니다.
7. `idempotency_key`: 외부 저장 API 호출에 같은 요청이 두 번 들어가도 한 번만 반영되게 합니다.

## 9. 이번 확인에서 본 근거

- `skills/search/SKILL.md`는 Valuehire AI Search의 절차 문서이며, ClickUp 포지션 intake → 해석 → 검색 전략 → 후보 발굴 → 평가 → 정규화 → handoff 흐름을 정의합니다.
- Hermes Gateway의 `gateway/run.py`는 세션별 `_running_agents`를 추적하고, 실행 시작 직후 `_AGENT_PENDING_SENTINEL`로 동시 진입을 막습니다.
- 같은 세션에서 실행 중 새 메시지가 오면 `/queue`, `/steer`, `/stop`, busy input mode에 따라 큐잉·스티어·인터럽트로 처리합니다.
- 하위 에이전트가 실행 중이면 인터럽트가 작업을 날릴 수 있어 큐잉으로 보호하는 로직이 있습니다.

## 10. SOT 반영 권장 규칙

네, SOT(단일 진실 원천 — 운영 기준을 한 곳에서 정하는 문서)에 “연속 입력은 순차 처리”를 명시할 수 있습니다. 오히려 Search 운영에서는 이 규칙을 SOT로 박아두는 편이 안전합니다.

권장 SOT 문구는 다음입니다.

```text
Valuehire AI Search의 Discord 동일 세션 연속 입력은 기본적으로 순차 처리한다.
같은 Discord DM/채널/스레드에서 Search 또는 Multisearch 요청이 연속으로 들어오면 새 병렬 실행을 만들지 않고 FIFO 큐에 넣는다.
진행 중 요청의 조건 보강은 /steer를 사용하고, 독립된 다음 작업 예약은 /queue를 사용한다.
사용자가 명시적으로 /stop 또는 cancel 의도를 보낸 경우에만 현재 실행을 중단한다.
외부 저장/ClickUp 반영/포털 브라우저 사용/live 발송이 포함된 작업은 position_lock, portal_lock, candidate_dedupe_key를 통과해야 한다.
```

운영 정책으로는 다음처럼 나누는 것이 좋습니다.

- 같은 Discord 세션 + 일반 Search 요청 연속 입력: FIFO 순차 처리
- 같은 Discord 세션 + `/queue`: FIFO 순차 처리, 메시지 병합 금지
- 같은 Discord 세션 + `/steer`: 현재 실행에 조건 추가, 새 Search 실행 아님
- 같은 Discord 세션 + `/stop`: 현재 실행 중단 후 잠금 해제
- 서로 다른 세션: 병렬 실행 가능하되 포털/저장소 lock은 공유
- live 저장/발송: 항상 dry-run 선행, owner 승인 뒤 실행

구현 설정 관점에서는 Hermes Gateway의 busy input 기본값을 `queue`로 두는 것이 SOT와 가장 잘 맞습니다. 단, SOT 문구를 문서에 넣는 것만으로 런타임 동작이 바뀌지는 않습니다. 실제 적용에는 Hermes 설정의 `display.busy_input_mode`를 `queue`로 바꾸고 Gateway 재시작/리로드 뒤 Discord에서 연속 입력 테스트를 해야 합니다. 현재 확인 기준으로 `~/.hermes/config.yaml`의 값은 `busy_input_mode: interrupt`였으므로, 아직 “같은 세션 무조건 순차 처리”가 강제 적용된 상태는 아닙니다.

## 11. 최종 결론

사장님이 여러 쉘 창에서 Search Skill을 동시에 주입하는 것은 “스킬 주입” 자체로는 문제 없습니다. 하지만 실제 후보 서치 운영에서는 외부 포털, 후보 저장소, ClickUp, Discord 응답 채널이 공유 자원이므로 병렬 실행 정책이 꼭 필요합니다.

가장 안전한 운영 원칙은 다음입니다.

- 여러 포지션을 의도적으로 한 번에 돌릴 때: Multisearch
- 같은 Discord 세션에서 다음 일을 예약할 때: `/queue`
- 진행 중인 작업에 조건을 추가할 때: `/steer`
- 같은 포지션 저장/발송이 걸린 작업: position lock + dedupe + owner 승인
- 단순 조사 병렬화: 쉘 여러 개 가능, 단 결과 경로와 브라우저 세션 분리
