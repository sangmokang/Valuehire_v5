Search Skill 동시 주입/동시 호출 동작 논리

작성일: 2026-06-10 · 대상: 여러 쉘 창, Discord, Search Skill, Multisearch
결론부터 말씀드리면, Search Skill 파일 자체가 서로 섞이는 일은 없습니다. 다만 같은 세션에서 동시에 들어온 요청은 Hermes Gateway의 실행 가드에 따라 큐잉, 인터럽트, 스티어 중 하나로 처리됩니다.

핵심 비유

Search Skill은 레시피 파일이고, 각 Hermes 실행은 주방입니다. 여러 주방이 같은 레시피를 동시에 읽을 수는 있지만, 같은 냄비에 동시에 재료를 넣으면 순서 정리 규칙이 필요합니다. Discord의 같은 대화방/같은 세션이 그 “같은 냄비”입니다.

용어 정리

Skill 주입

skills/search/SKILL.md 같은 절차 문서를 이번 실행의 업무 규칙으로 넣는 것입니다.

세션

Hermes가 대화 맥락을 묶는 단위입니다. Discord에서는 플랫폼, 채널, 스레드, 사용자 설정 조합으로 정해집니다.

Gateway

Discord나 Telegram 메시지를 Hermes 에이전트 실행으로 연결하는 중간 서버입니다.

큐잉

현재 작업이 끝난 뒤 다음 차례로 처리하도록 쌓아두는 방식입니다.

인터럽트

현재 작업을 중단시키고 새 메시지를 우선 처리하는 방식입니다.

스티어

현재 작업을 끊지 않고 다음 도구 호출 뒤에 새 지시를 끼워 넣는 방식입니다.

1. 여러 쉘 창에서 동시에 Search Skill을 주입하면

각 쉘 창에서 Hermes를 별도 프로세스로 실행하면 기본적으로 각각 독립된 실행 컨텍스트를 가집니다.

각 프로세스가 Skill 읽기→ 각자 프롬프트에 주입→ 각자 모델/도구 실행→ 각자 결과 생성

단순 주입은 읽기 작업이므로 Skill 파일 내용이 서로 합쳐지거나 오염되지 않습니다. 위험은 Skill이 아니라 공유 자원에서 생깁니다.

같은 결과 파일, 같은 브라우저 프로필, 같은 포털 로그인 세션, 같은 ClickUp/DB 저장소를 동시에 쓰면 덮어쓰기·중복 저장·검색 조건 꼬임이 생길 수 있습니다.

2. Discord 같은 방에 여러 번 입력하면

Gateway는 세션별로 실행 중인 에이전트를 추적합니다. 같은 세션에 이미 실행 중인 에이전트가 있으면 새 에이전트를 무조건 하나 더 띄우지 않고, busy-input 정책에 따라 처리합니다.

  • _AGENT_PENDING_SENTINEL로 실행 시작 직후의 짧은 동시 진입 틈을 막습니다.
  • _running_agents로 세션별 실행 중인 에이전트를 추적합니다.
  • /queue는 먼저 들어온 순서대로 다음 턴을 만듭니다.
  • /steer는 현재 실행 중 다음 도구 호출 뒤에 지시를 끼워 넣습니다.
  • /stop은 현재 세션 실행을 끊고 잠금을 풉니다.
모드동작Search 운영 의미
queue현재 작업 뒤에 처리가장 안전합니다. Search를 순서대로 처리합니다.
interrupt현재 작업 중단 후 새 요청 우선진행 중 후보 수집이 끊길 수 있습니다.
steer현재 작업에 지시 추가“조건 하나 더 추가”에 적합합니다.

3. 서로 다른 Discord 채널/스레드/DM이면

세션 키가 다르면 별도 실행처럼 병렬로 처리될 수 있습니다. 예를 들어 사장님 DM, 다른 채널, 다른 스레드가 서로 다른 세션으로 잡히면 각각의 _running_agents 엔트리가 생깁니다.

다만 중앙 Gateway, 모델 API 한도, 포털 브라우저 세션, ClickUp API 같은 자원은 공유될 수 있으므로 병목과 충돌은 여전히 관리해야 합니다.

4. Multisearch와 임의 동시 입력의 차이

Multisearch는 여러 포지션을 처음부터 하나의 배치로 다루는 방식입니다. 임의 동시 입력은 독립 요청 여러 개가 거의 동시에 들어오는 것입니다.

구분장점위험
Multisearch우선순위, 중복 제거, 결과 포맷, 저장 위치를 통제하기 쉽습니다.초기 요청을 묶어 정리해야 합니다.
임의 동시 입력운영자가 빠르게 요청을 던질 수 있습니다.중복 서치, 중복 저장, 인터럽트, 브라우저 충돌 위험이 큽니다.

5. 권장 운영 원칙

  1. 여러 포지션을 의도적으로 돌릴 때는 Multisearch를 우선 사용합니다.
  2. Discord에서 다음 일을 예약할 때는 /queue <요청>을 사용합니다.
  3. 진행 중인 Search에 조건만 보탤 때는 /steer <조건>을 사용합니다.
  4. 잘못 돌기 시작한 작업은 /stop으로 끊고 새로 요청합니다.
  5. 저장/발송 작업은 포지션별 잠금과 후보 중복키를 둡니다.
  6. 기본은 dry-run으로 두고, live 저장·발송은 명시 승인 뒤 처리합니다.

6. 구현 가드 제안

search_run_id

실행마다 고유 ID를 붙입니다.

position_lock

같은 포지션 저장 작업은 한 번에 하나만 허용합니다.

portal_lock

같은 포털 브라우저 세션은 한 작업만 잡게 합니다.

candidate_dedupe_key

프로필 URL, 이메일, 이름+회사 등으로 중복을 막습니다.

result_namespace

결과 파일명에 날짜, 포지션, 실행 ID를 넣습니다.

idempotency_key

같은 외부 저장 요청이 두 번 와도 한 번만 반영되게 합니다.

7. 최종 판단표

상황예상 동작권장
여러 쉘 창에서 Search Skill 동시 주입각 프로세스 독립 실행결과 파일, 브라우저, 저장소 분리
Discord 같은 DM에 연속 입력같은 세션 busy 정책 적용/queue 또는 /steer 명시
Discord 다른 채널/스레드 입력세션이 다르면 병렬 가능포털/저장소 충돌 주의
Multisearch 사용하나의 배치로 통제여러 포지션이면 가장 권장
live 저장/발송 포함중복·충돌 가능승인, 중복키, dry-run 선행

근거

  • skills/search/SKILL.md: ClickUp 포지션 intake, 해석, 검색 전략, 후보 발굴, 평가, 정규화, handoff 흐름을 정의합니다.
  • gateway/run.py: _AGENT_PENDING_SENTINEL, _running_agents, /queue, /steer, /stop, busy-input 처리를 통해 같은 세션 동시 입력을 제어합니다.
  • 하위 에이전트 실행 중에는 인터럽트가 작업을 날릴 수 있어 큐잉으로 보호하는 로직이 있습니다.

8. SOT 반영 권장 규칙

네, SOT(단일 진실 원천 — 운영 기준을 한 곳에서 정하는 문서)에 “연속 입력은 순차 처리”를 명시할 수 있습니다. Search 운영에서는 이 규칙을 SOT로 박아두는 편이 안전합니다.

Valuehire AI Search의 Discord 동일 세션 연속 입력은 기본적으로 순차 처리한다.
같은 Discord DM/채널/스레드에서 Search 또는 Multisearch 요청이 연속으로 들어오면 새 병렬 실행을 만들지 않고 FIFO 큐에 넣는다.
진행 중 요청의 조건 보강은 /steer를 사용하고, 독립된 다음 작업 예약은 /queue를 사용한다.
사용자가 명시적으로 /stop 또는 cancel 의도를 보낸 경우에만 현재 실행을 중단한다.
외부 저장/ClickUp 반영/포털 브라우저 사용/live 발송이 포함된 작업은 position_lock, portal_lock, candidate_dedupe_key를 통과해야 한다.
입력 상황SOT 처리 원칙
같은 Discord 세션 + 일반 Search 연속 입력FIFO 순차 처리
같은 Discord 세션 + /queueFIFO 순차 처리, 메시지 병합 금지
같은 Discord 세션 + /steer현재 실행에 조건 추가, 새 Search 실행 아님
같은 Discord 세션 + /stop현재 실행 중단 후 잠금 해제
서로 다른 세션병렬 가능하되 포털/저장소 lock은 공유
live 저장/발송항상 dry-run 선행, owner 승인 뒤 실행
구현 설정 관점에서는 Hermes Gateway의 busy input 기본값을 queue로 두는 것이 SOT와 가장 잘 맞습니다. 단, SOT 문구를 문서에 넣는 것만으로 런타임 동작이 바뀌지는 않습니다. 실제 적용에는 Hermes 설정의 display.busy_input_mode를 queue로 바꾸고 Gateway 재시작/리로드 뒤 Discord에서 연속 입력 테스트를 해야 합니다. 현재 확인 기준으로 ~/.hermes/config.yaml의 값은 busy_input_mode: interrupt였으므로, 아직 “같은 세션 무조건 순차 처리”가 강제 적용된 상태는 아닙니다.
