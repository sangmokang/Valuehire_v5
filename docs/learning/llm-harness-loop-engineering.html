오리엔테이션: 왜 하니스인가

LLM은 같은 입력에 다른 출력을 낸다. 이 변동성(variance)이 LLM 시스템 품질 문제의 뿌리다. 하니스 엔지니어링의 목적은 docs/harness.md:3에 한 문장으로 박혀 있다:

"목적은 단 하나: 출력의 변동성을 줄여 재현 가능하게 만든다."

모델을 더 똑똑하게 만드는 게 아니라, 모델을 둘러싼 구조물(게이트·검증·루프)로 출력의 분산을 좁힌다. 이 저장소의 하니스는 문서가 아니라 배관이다:

게이트명령기계 강제 지점
0 시작자격make red-ledgerscripts/harness/red-ledger.sh — 미해결 RED 있으면 exit 1, 새 작업 금지
2 격리make task NAME=xscripts/harness/task.sh — worktree 강제, main 읽기 전용
4a 검증./verify.shverify.sh — pytest 단일 수렴, exit 0만 GREEN
5 배송make shipscripts/harness/ship.sh — main이면 즉시 거부, verify 재실행 후 PR
CIpush/PR.github/workflows/verify.yml — 초록 없이 merge 불가
핵심 관점. "게이트를 늘리지 않는다"가 설계 원칙이다 (docs/harness.md:101 단일 verify 게이트 원칙). 스킬·문서·배관이 늘어도 검사는 전부 ./verify.sh 하나로 수렴한다. 검증 통로가 하나면 우회로가 없다.

학습 테마

  • Anthropic Building Effective Agents — 단순성·투명성·도구 설계(ACI). 우리 하니스와의 정렬표는 docs/harness-engineering.md §5.
  • 워크플로 vs 에이전트: 정형 작업은 게이트 워크플로로, 모델 판단이 필요한 곳만 자율에 맡긴다.

LLM 호출의 계약화

이 저장소가 LLM을 부르는 곳은 놀랄 만큼 적다. 부르는 곳마다 입출력 계약이 있고, 계약 위반의 종착지는 언제나 "예외 → 차단"이지 "대충 통과"가 아니다.

1.1 — LLM 클라이언트는 함수 하나다

왜: 호출 지점이 흩어지면 비용·모델·에러 처리가 제각각이 된다.

실제 코드

tools/multi_position_sourcing/llm_keywords.py:329 — 운영용 클라이언트는 claude -p 서브프로세스를 감싼 클로저다:

def claude_keyword_client(*, model: str = "haiku", ...) -> LLMClient:
    """운영용 LLM 클라이언트 — 로컬 claude -p 호출(비용 헌법: 저가 모델 기본)."""
    def _call(prompt: str) -> str:
        if shutil.which("claude") is None:
            raise KeywordGenerationError("claude CLI를 찾지 못했습니다.")
        result = run_command(["claude", "-p", "--model", model, prompt], ...)
        if result.returncode != 0:
            raise KeywordGenerationError(f"claude -p 실패(rc={result.returncode}): ...")
        return result.stdout or ""
    return _call
  • 비용 헌법: 기본 모델이 haiku. 키워드 생성 같은 기계적 작업에 비싼 모델을 쓰지 않는다.
  • 주입 가능(run_command 인자): 테스트는 진짜 LLM 없이 fake로 조립·파싱을 검증한다.
  • API 키 대신 로컬 CLI(Max 구독) — 돈이 새는 경로 자체를 만들지 않는다.

학습 테마

  • 의존성 주입으로 LLM을 테스트 가능하게 만들기 · LLM 게이트웨이 패턴 · 모델 티어 라우팅(싼 모델 기본, 비싼 모델은 근거 있을 때만)

1.2 — 구조화 출력: 파싱이 아니라 검문

왜: LLM의 JSON은 "아마도 JSON"이다. 믿는 순간 시스템이 LLM의 변동성을 물려받는다.

실제 코드

tools/multi_position_sourcing/llm_keywords.py:78의 _extract_json_object + :95의 _clean_keywords:

def _clean_keywords(raw_keywords) -> tuple[str, ...]:
    if not isinstance(raw_keywords, list):
        raise KeywordGenerationError('"keywords"가 리스트가 아닙니다.')
    seen: set[str] = set(); ordered: list[str] = []
    for item in raw_keywords:
        if not isinstance(item, str): continue
        kw = item.strip()
        if not kw or kw in seen: continue
        seen.add(kw); ordered.append(kw)
    if not ordered:
        raise KeywordGenerationError("LLM이 유효한 키워드를 하나도 내지 않았습니다 (0건 검색 위험).")
    return tuple(ordered)

검문 4단: ① 최상위가 객체인가 ② 필드가 리스트인가 ③ 원소별 타입·공백·중복 정리 ④ 정리 후 0건이면 예외. LLM 출력은 신뢰 경계(trust boundary) 밖의 입력으로 취급된다 — 사용자 입력과 똑같이.

Loop Engineering I: 순회 루프

루프 엔지니어링의 첫 질문은 "어떻게 도는가"가 아니라 "언제, 왜 멈추는가"다. 중단 조건이 없는 루프는 봇처럼 굴다가 차단당하거나(SOT 2 위반), 영원히 돈다.

  • 계약 위반 잡은 실행 전에 걸러 failed로 회수 — 나쁜 입력이 LLM까지 가지 않는다.
  • idle일 때 스스로 일감을 1건 만든다(_enqueue_idle_variant) — 심야에 미소진 검색 변형을 소진하는 자기 급유 루프. 단 "회당 1건"으로 스로틀.

5.1 — evaluate_send: 사유 0개일 때만 allowed

실제 코드

tools/multi_position_sourcing/auto_send.py:238 — SOT28(docs/sot/28-auto-send-policy.json)의 유일한 발송 관문:

# 점수: int 만 인정(bool 금지). None=미채점, 그 외 타입(float/NaN/str)=오염 — 전부 차단.

하니스 조이기 II: Generate-Verify 분리

하니스의 심장부. 불변식은 하나다 — 만든 자는 검증하지 않는다. 생성자(G)가 만들고, 다른 도구(V1=Codex)가 깨고, 기억을 리셋한 검증자(V2=Claude)가 그 판정마저 깬다. 세 역할이 계약(T) 기준에서 일치해야 통과다.
