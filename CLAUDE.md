# CLAUDE.md — 이 저장소의 가장 윗 규칙 (SOT)

> 이 문서가 제일 위에 있는 규칙입니다. 다른 어떤 문서나 습관보다 먼저 따릅니다.
> 여기 적힌 **SOT 불변식**은 어떤 상황에서도 약하게 바꾸지 않습니다.

## ⛔ 0번 규칙 — 사장님에게 말할 때 (SOT 불변식 · 절대 약화 금지)

**한국어로, 쉽게, 짧게 말한다.**

- CLI(터미널) 창에서는 **어려운 기술 용어를 쓰지 않는다.**
- 영어 약자·내부 코드명·복잡한 절차 이름을 그대로 늘어놓지 않는다.
- 꼭 필요한 전문 용어가 있으면, 바로 옆에 **한 줄로 쉬운 말**을 붙인다.
- "무엇을 했는지 / 왜 / 다음에 뭘 할지"만 사람 말로 알려준다.

| 이렇게 말하지 말 것 (X) | 이렇게 말할 것 (O) |
|---|---|
| "Gate 4a verify exit 0, 393 passed" | "검사 다 통과했어요(393개). 이상 없습니다." |
| "worktree에서 RED→GREEN 후 ship" | "따로 작업방 만들어서 고치고, 올렸습니다." |
| "PR mergeable, CI green" | "올린 거 합쳐도 되는 상태예요. 검사도 통과했고요." |

> 한 줄 요약: **사장님은 코드 전문가가 아니다. 결과를 쉬운 한국어로 전한다.**

## SOT 불변식 — 운영 규칙 (절대 약화 금지)

1. **3사(사람인·잡코리아·링크드인) 로그인은 내(자동화)가 무조건 한다 — 절대 막지 않는다.** 저장된 자격증명으로 자동 로그인·재로그인을 항상 수행하고, 이를 막는 코드·규칙이 있으면 SOT 위반이므로 삭제한다. **오직 2FA(본인확인)·캡차·체크포인트가 뜰 때만 사장님이 직접 처리**하며, 그 순간에는 **브라우저 창을 앞으로 띄워** 사장님이 바로 볼 수 있게 한다(그 외에는 사람에게 넘기지 않는다). 상세 기계 명세는 `docs/sot/26-portal-login-spec.json`(INV1·INV2·INV6).
2. **사장님이 크롬을 쓰는 동안에는 그 컴퓨터의 자동 작업이 잠깐 멈췄다가, 사장님이 손을 떼면 자동으로 다시 이어서 한다(잠깐 양보·자동 재개, R4).** 구체 기준(2026-07-20 사장님 지시로 개정, SOT29 INV9): **사장님 개입으로 보는 것은 크롬에서 3사(사람인·잡코리아·링크드인) 화면을 만질 때뿐이다(유튜브 등 다른 화면은 개입 아님 — 자동 작업 계속). 3사 개입·이상 신호(캡차·2FA) 후 1분(60초) 동안 이상이 없으면 자동 재개하며, 로그인은 우선순위가 매우 높다.** 멈추고 방치하지 않는다 — 반드시 자동으로 재개하며, 자동 재개를 영구 차단하는 코드(작업 목록 폐기·무기한 중단·10분 이상 고정 대기)는 SOT 위반이므로 발견 즉시 삭제한다. 그리고 어느 때든 **봇처럼 굴지 않는다**: 혼자 창을 열었다 닫았다 반복, URL 연속 입력, 알람 뜬 뒤 같은 시도 무한 반복 같은 기계적 행동을 하지 않고, 막히면 원인을 해결한 뒤 사람처럼 움직인다.
3. **제안·메일 "보내기"는 자동발송 정책(`docs/sot/28-auto-send-policy.json`, SOT28)의 게이트를 전부 통과한 경우에만 자동으로 누른다.** (2026-07-07 사장님 명시 지시로 개정) 게이트 = 적합도 85점 이상 + 하드제외 없음 + 기계 검문(precheck) 통과 + 글자수 캡 이내 + 90일 내 중복 아님 + 일일 상한 미만 + 킬스위치 꺼짐. **하나라도 어긋나면 발송하지 않고(fail-closed), 게이트 밖 자율 발송은 지금처럼 사람이 마지막에 누른다.** 모든 자동 발송은 원장에 기록한다. **단, 사장님이 특정 메일/발송을 직접 "보내라·발송하라"고 지시하면 그 건은 자율 발송이 아니라 사장님 명시 지시이므로, 초안에서 멈추지 않고 Send까지 눌러 실행한다**(발송 도구가 초안만 지원하면 그 한계를 알린다).
4. **(0번 규칙) 사장님께는 한국어로 쉽게 설명한다.** ← 말하기 방식도 SOT다.
5. **내가 만든 코드는 믿지 않는다 — 두 번 깐다. 그리고 시작 전에 과거 지시를 먼저 찾는다.**
   - 내 코드는 늘 헛점이 있다고 전제한다. **작은 기능 단위로** 만들고,
     (1) **내가 먼저 스스로 적대적으로 깨본 뒤**(반증 시도),
     (2) **Codex Rescue에게 넘겨 한 번 더 깐다**(독립 2차 적대검증).
     두 패스를 다 통과하기 전엔 사장님께 "됐다"고 가져오지 않는다.
   - **코딩을 시작할 때마다**, 사장님이 **전에 시킨 것**이 이미 코드·문서·스킬·메모리에
     있는지 먼저 찾아본다(중복 구현 금지, 과거 지시 회수).
   - 절차는 `docs/harness.md`의 게이트(과거 지시 회수 + 게이트 4b 2패스)로 강제한다.

이 다섯 가지는 코드·문서·작업 어느 단계에서도 약하게 바꾸지 않는다.

## 5번 규칙 풀이 — "왜 두 번 까는가" (사람 말로)

사장님은 제가 만든 코드를 믿지 않으십니다. 맞습니다 — 제 코드는 늘 헛점이 있을 수 있습니다.
그래서 일을 이렇게 합니다.

1. **작게 자른다.** 한 번에 큰 덩어리를 만들지 않고, 작은 기능 하나씩 만듭니다.
2. **제가 먼저 깹니다.** 만든 걸 칭찬하지 않고, 일부러 깨뜨리려고 공격해 봅니다
   (빈 값, 잘못된 입력, 막힌 사이트, 중복 같은 나쁜 경우를 먼저 넣어 봅니다).
3. **Codex Rescue가 한 번 더 깹니다.** 저와 다른 눈(다른 도구)으로 같은 코드를 또 공격합니다.
   둘 다 못 깨야 통과입니다.
4. **시작 전에 과거를 봅니다.** "전에 사장님이 이거 시키신 적 있나? 이미 코드에 있나?"를
   먼저 찾습니다. 같은 걸 또 만들지 않기 위해서입니다.

> 한 줄 요약: **작게 만들고 → 내가 깨고 → Codex Rescue가 또 깨고 → 그제야 가져온다.
> 시작 전엔 과거 지시부터 찾는다.**

## 작업 방식

- 작업 절차(과거 지시 찾기 → 검사 먼저 만들기 → 고치기 → 두 번 깨기 → 확인 → 올리기)는
  `docs/harness.md`를 따른다.
- `main`은 직접 건드리지 않는다. 따로 작업방(worktree)을 파서 거기서 한다.
- 무엇이든 끝났다고 말하기 전에 **실제로 검사를 돌려보고** 결과를 보고한다.
- 전체 코드를 다시 훑어볼 때는 `docs/prompts/goal-full-codebase-review.md`(Goal 프롬프트)를 쓴다.
- 3대 머신(맥미니·맥북·윈도우) 함대 운영·Discord 명령·계정↔머신 바인딩은 `docs/sot/29-fleet-control.md`(+`.json`)를 SOT 로 따른다.
- Discord 단일 direct gateway 전환과 Hermes 격리·코드 제거·비밀 회전·최종 폐기는
  `docs/sot/33-hermes-retirement.md`를 SOT 로 따른다. HR-0~HR-7을 한 작업방에 합치지 않는다.
